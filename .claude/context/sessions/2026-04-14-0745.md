Key research findings that drove the design:
1. **Anthropic MCP best practices**: deterministic predicates belong in stable server code, not the LLM reasoning loop. So `risk_check` and `validate_signal` are the right things to ship now (no LLM); the model-bound stubs (`generate_signal`, `run_single_feature_test`) are the right things to defer.
2. **QuantConnect Algorithm Framework**: the canonical layered pattern is Alpha -> PortfolioConstruction -> RiskManagement -> Execution. `validate_signal` and `risk_check` are *complementary*, not redundant: validate looks at the signal alone, risk_check looks at the signal vs the current portfolio.
3. **FINRA 15c3-5 + FIA whitepaper**: canonical risk-check evaluation order is regulatory-fatal -> financial-fatal -> soft, fail-fast on hard violations. I used: schema -> SELL position consistency -> daily count -> per-ticker concentration -> total exposure -> cash floor -> drawdown circuit breaker.
4. **TOCTOU between validate and execute**: known acceptable risk for paper trading. Documented in the docstring; the eventual `publish_signal` (Phase 4.1) must re-check at commit time.
5. **No cross-server calls**: explicit MCP anti-pattern. signals_server stays self-contained.
