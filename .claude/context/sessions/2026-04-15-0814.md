### Key findings

- **The Cycle 5 deferral was incorrect**. "After Storage Write
  API migration" conflated the streaming-buffer DML trap (which
  blocks UPDATE/DELETE/MERGE) with the streaming-buffer as a
  general restriction on writes. In fact, `insert_rows_json`
  always allows new row inserts, even on a table that has other
  rows in the streaming buffer. Append-only is orthogonal to
  the trap. This cycle corrects the deferral and completes the
  4.2.4 durable persistence surface without the Storage Write
  API migration being a prereq. The Storage Write API is a
  meaningful future cycle (unlocks DML on recently-streamed
  rows, 50% cost savings on bulk inserts), but it's a
  replacement tech, not a prereq for append-only logging.

- **The dedicated qa-evaluator subagent + pre-baked assertion
  block continues to work reliably**. 34 checks, 3 tool uses,
  69 seconds, zero retries, first Phase-4 cycle to hit perfect
  10/10/10/10/10 on the first attempt. This is the fourth
  consecutive cycle (4.2.3.2, 4.2.3.3, 4.2.4, 4.2.4.2) to use
  this pattern successfully without `Stream idle timeout`
  issues.

- **Helper-method refactor was worth the 48 lines.** The Cycle 5
  publish path inlined its 30-line BQ wire-up inside
  `_append_signal_history`. This cycle chose to extract a helper
  `_save_outcome_event_to_bq` instead of inlining a 30-line
  block 3 times. Reason: 3 inlined blocks would have been ~90
  lines in `track_signal_accuracy` alone, harder to read and
  harder for QA to audit. The 44-line helper is called 3x from
  one-line call sites, for a total of ~47 lines including call
  sites. Simpler for QA, simpler for future-Ford, and isolates
  the BQ surface to one place (easy to stub in tests).

- **`formatters.py` non-ASCII is NOT a security issue.** The 5
  non-ASCII bytes in `backend/slack_bot/formatters.py` are
  Unicode BULLET + EM DASH in Slack Block Kit user-facing
  message text. Zero `logger.*()` calls in the file (confirmed
  by grep). The `.claude/rules/security.md` ASCII-only rule
  scopes to logger calls only (Windows cp1252 uvicorn handler
  crash). Block Kit rendering preserves Unicode correctly
  across all client platforms; normalizing these bytes would
  be a UX regression. Documented here so future-Ford doesn't
  waste a cycle on this false-positive.
