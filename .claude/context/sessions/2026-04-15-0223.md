1. **Use `insert_rows_json`** -- matches the existing `save_outcome`
   and `save_report` precedent in `BigQueryClient`. Storage Write
   API would give 50% cost savings + DML compatibility but requires
   protobuf schemas + protoc; deferred to a follow-up cycle.
2. **No DML UPDATE in this cycle.** Streaming buffer holds rows for
   30-90 minutes during which UPDATE/DELETE/MERGE/TRUNCATE fail with
   the documented "rows in streaming buffer" error. The outcome-update
   path is the natural casualty. Defer until Storage Write API or
   event-log design is in place.
3. **Idempotent CREATE TABLE pattern** via `try: client.get_table /
   except: client.create_table` -- byte-structurally identical to the
   existing `migrate_paper_trading.py`.
4. **Append-only event-log schema** per quant audit-trail best
   practice. Each row has `event_kind` ("publish" this cycle,
   "outcome" in a future cycle). NEVER UPDATE an existing row;
   instead append a NEW row with the post-scoring state.
5. **17-field schema**: identity (signal_id/ticker/signal_type),
   prediction state (confidence/signal_date/entry_price/factors_json/
   created_at), outcome placeholders (outcome/scored/hit/exit_price/
   exit_date/forward_return_pct/holding_days), audit metadata
   (recorded_at/event_kind). The `date` field is renamed `signal_date`
   in BQ to avoid the BQ `DATE` reserved-word collision.
6. **Never-raise boundary**: `_append_signal_history` wraps the BQ
   call in `try: ... except Exception as e: logger.warning(f"bq_signal_log
   save failed: {type(e).__name__}")`. ASCII only per security.md.
   The outer `publish_signal` step 9 try/except is also there as
   defense in depth.
7. **In-memory append happens first, BQ call second**. The in-memory
   state is the canonical source of truth this cycle; BQ is the
   durable read-after-restart path for a future cycle.

### Key findings

- **Storage Write API is the right long-term choice** for the
  outcome-update path. The 30-90 minute streaming buffer DML window
  is a hard blocker for any UPDATE-based design on a table that
  receives streaming inserts. The `insert_rows_json` precedent in
  `BigQueryClient` is a legacy choice this cycle inherits and
  preserves; a Phase 4.2.4-follow-up cycle should adopt the
  Storage Write API across all four `BigQueryClient` insert
  methods (`save_report`, `save_outcome`, `save_signal`, plus the
  paper-trading insert methods).

- **Append-only event log over UPDATE-in-place is the correct
  shape for trading audit trails**. Industry research (Yukti,
  LEAN, QuantConnect) consistently recommends this. The `event_kind`
  + `recorded_at` fields are the foundation. The future outcome-event
  cycle will append a SECOND row keyed by the same `signal_id`
  with `event_kind="outcome"`, and a window function in the read
  path will compute the latest state per signal_id. Never UPDATE.

- **The dedicated qa-evaluator subagent + pre-baked assertion block
  is the reliable path** for QA. 42 checks, 5 tool uses, ~110s,
  no timeout. Confirms the 2026-04-14-2026 + 2026-04-14-2245
