   fallback naturally tolerates `"  2026-04-01  "` and `"2026-4-1 "`

   whitespace is never a chronological ambiguity. Updated contract

   lead-self assertion.

   set. No `OverflowError` needed for `int()` coerce into
   `date.fromisoformat`.
4. **Helper is a `@staticmethod` on `SignalsServer`**, not module-
   level, matching the file's "everything on the class" shape.
5. **No dateutil / pendulum / arrow / third-party date libs** --
   stdlib-only per no-new-deps rule and Phase 4.2.3.1 `math`
   precedent.
6. **Never raise from a read API** -- both the helper and the
   call-site return `None` on every failure path.
7. **Boundary semantics are inclusive** (`>=`, not `>`), preserved
   from pre-fix behavior.
8. **Fallback normalization: f-string `{int(x):02d}`** -- folds
   parse + pad into one step, gives natural reject path via
   `int()` ValueError on non-numeric tokens.

- **SN4 was a real bug with concrete failure cases**. Contract SC13
  documents: `since_date="2026-4-1"` with stored `"2026-04-15"` is
  False under lex compare (index-5 "0" < "4") -- a valid April 15
  record is spuriously excluded. SC14 documents the reverse: an
  unpadded January record spuriously passes a padded April filter.
  Both cases now return correct chronological results.

- **`int()` whitespace stripping is a happy accident**. My initial
  contract assumed ADV4/ADV5 (whitespace-padded inputs) would
  return None, but the split path succeeds because `int("  2026 ")`
  strips whitespace. I updated the contract to document this as
  the expected behavior (lenient but safe) rather than a bug. This
  is the one contract correction I made during GENERATE.

- **Python 3.11+ basic ISO acceptance**: `date.fromisoformat("20260401")`
  now returns `date(2026,4,1)` on the strict path. Documented as
  expected ADV3 behavior. Behaviorally benign -- any application
  that deliberately passes `"20260401"` to mean "April 1, 2026"
  gets the right answer.
