### Suggested follow-up: parallel-race mitigation

Three documented races now (Phase 4.2.2 EVALUATE, Phase 4.2.3.3 here,
plus one earlier 4.2.2 GENERATE). Pattern: both Fords arrive within
30 minutes of each other, read the same prior session's "Next run
should" list, pick the same top item, do the same work. Mitigation
options worth raising with Peder:

1. **Session-start advisory lock** -- a 1-line check at startup that
   greps `git log --since='30m' origin/main` for the candidate phase
   ID and skips it if found. Soft, not enforced. Would have avoided
   today's race entirely.
2. **Lockfile in `handoff/current/`** -- session writes its claimed
   step ID + start timestamp to `handoff/current/.lock`, next session
   reads it before picking work. More robust but adds a new file.
3. **Round-robin "Next run should" item selection** -- session N picks
   item (N mod K) from the list. Deterministic dispersal but requires
   sessions to know N. Probably overkill.
4. **Accept the duplication** -- micro-fixes are cheap, cross-validation
   is a feature not a bug, no harm done. Status quo.

I'd rank: option 1 > 4 > 2 > 3. Not a session-blocker, raising for
your awareness.
