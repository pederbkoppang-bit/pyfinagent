# Findings — phase-24.4 — Agent Topology + Per-Agent Rationale Flow

## Executive summary

The byte-identical Trader/RiskJudge rationale bug is structural, not cosmetic. The aliasing occurs in `backend/services/autonomous_loop.py:719` — the lite-path `_run_claude_analysis` makes ONE LLM call producing ONE `reason` string, then writes it to BOTH `analysis.reason` (consumed by Trader) and `risk_assessment.reason` (consumed by RiskJudge). No independent risk-specific LLM call exists. A patch at `backend/services/signal_attribution.py:131-154` detects the byte-identical pair (weight=0.0 + rationale match) and displays an amber "lite-path" badge — this is cosmetic, not a fix. The sparse-drawer symptom (3 of ~20 agents) is downstream of bucket 24.2 (pipeline routing): the lite path is the default, and lite mode produces only 2-3 signal rows. The full 28-skill Layer-1 pipeline exists in code but its individual skill rationales (NLP sentiment, scenario weights, bias detector outputs) are never persisted into the `signals` JSON column even when the full pipeline runs. The hypothesized file location (`multi_agent_orchestrator.py`) was wrong — that file is the Slack/iMessage Layer-2 orchestrator and does NOT drive paper-trading rationale. Six phase-25 candidates close the gap: decouple RiskJudge with an independent LLM call, remove the cosmetic patch after the structural fix, surface Layer-1 skill outputs when full pipeline runs, normalize per-agent contribution weights, add "summary vs full" drawer toggle, and add a byte-identical regression test.

## Code-grounded findings

### F-1: The smoking gun — `autonomous_loop.py:719` lite-path aliasing

`backend/services/autonomous_loop.py:711-720`:

```python
return {
    ...
    "risk_assessment": {"reason": analysis["reason"]},  # <-- same string as Trader
    ...
    "full_report": {
        "analysis": analysis,  # analysis["reason"] is the Trader rationale
    },
}
```

The lite-path LLM call at `autonomous_loop.py:670-690` produces ONE structured response with ONE `reason` field. It is then written to both `analysis.reason` (Trader's text) and `risk_assessment.reason` (RiskJudge's text). No second LLM call with a risk-specific prompt is made. Both fields are byte-identical to each other and to `full_report.analysis.reason`.

### F-2: Aliasing detection is cosmetic, not a fix

`backend/services/signal_attribution.py:117-157`:

```python
risk = analysis.get("risk_assessment") or {}
if isinstance(risk, dict):
    decision = risk.get("decision") or ""
    reasoning = (
        risk.get("reasoning")
        or risk.get("rationale")
        or risk.get("reason")  # <-- lite-path fallback

    ...
    risk_rationale = _trim(reasoning) or f"Decision: {decision}"

    trader_rationale_trimmed = _trim(trader_note) or f"Recommendation: {rec}"
    is_lite_dup = (
        risk_weight == 0.0
        and risk_rationale == trader_rationale_trimmed  # byte-identical detection

    # ... amber-badge substitution follows
```

The byte-identical detection at line 131-154 catches the aliasing pair (weight=0.0 + rationale match) and replaces the displayed text with `"Lite-path: Risk Judge inherited Trader's reasoning..."`. The `lite_path=True` flag is set; the frontend renders an amber badge. **This is a display patch, not a structural fix** — the RiskJudge still has weight=0.0 and contributes zero independent risk signal to the trade decision.

The lite path never populates `recommended_position_pct` (confirmed by `autonomous_loop.py:711-740` return dict shape). So `risk_weight` resolves to 0.0 every time the lite path runs. The display shows weight 0.00 (matching the operator screenshot).

### F-4: Sparse drawer — `3 of 20 agents` mathematics

From `backend/agents/_inventory.json` the topology has ~43 nodes across 4 layers:
- Layer 3 (harness): 3 agents (Main + Researcher + Q/A)
- Layer 2 (MAS): 7 agents
- Layer 1 (skills): 28+ agents
- Layer 4 (services): ~5 agents

The paper-trade drawer renders rows produced by `extract_signals_from_analysis` in `signal_attribution.py:57-157`. This function extracts at most 5 signal types: Analyst, Bull, Bear, Trader, RiskJudge.

**Lite path drawer coverage:**
- Analyst: absent (lite path produces no `analyst_summary`)
- Bull/Bear: absent (no debate step)
- Trader: 1 row
- RiskJudge: 1 row (with lite-path badge)
- Quant: 1 row IF a `candidate` dict is passed
- Total: **2-3 rows visible** — matching the operator's "3 of ~20" observation

**Full Gemini path drawer coverage** (when `settings.lite_mode = False`):
- Analyst + Bull + Bear + Trader + RiskJudge = 5 rows max
- Layer-1 28 skills: ZERO rows (individual skill outputs are never persisted into the `signals` JSON column even when the full pipeline runs)

So the drawer is sparse by architectural design, not just by lite-mode default. The Layer-1 28-skill surfacing is structurally absent.

### F-5: `multi_agent_orchestrator.py` is NOT the paper-trade path

The hypothesis incorrectly identified `backend/agents/multi_agent_orchestrator.py` as the aliasing site. Investigation confirms this file (1513 LOC) is the **Slack/iMessage Layer-2 orchestrator** with 4 agent types (COMMUNICATION, MAIN, QA, RESEARCH). It does NOT drive paper-trading decisions. The paper-trade flow is:

```
autonomous_loop.py::run_daily_cycle
  -> _run_single_analysis (full Gemini path) OR
  -> _run_claude_analysis (lite path)
  -> signal_attribution.extract_signals_from_analysis
  -> paper_trader.execute_buy
```

`multi_agent_orchestrator.py` is invoked only by the Slack bot for chat queries. The "Trader" and "RiskJudge" displayed in the drawer originate in `_run_claude_analysis` (lite) or `_run_single_analysis` (full), not in `multi_agent_orchestrator.py`. The bucket-spec canonical URL `multi_agent_orchestrator.py` is therefore a doc/spec anchor (referenced in the audit prompt) — the actual smoking gun lives in `autonomous_loop.py:719` + `signal_attribution.py:117-157`.

### F-6: Rationale endpoint trace

`backend/api/paper_trading.py:460-511` `GET /trades/{trade_id}/rationale`:
1. Reads BQ `paper_trades` row by `trade_id`
2. Loads the `signals` JSON column
3. Calls `group_signals_for_drawer(signals)` to produce a tree
4. Returns `{tree, raw, byte_identical_pairs}`

The frontend `frontend/src/components/AgentRationaleDrawer.tsx` (232 LOC) renders the tree with `lite_path` amber badges where flagged. The drawer knows about the flag but cannot independently inspect text — it relies on the backend's detection.

## External-research summary

### Source 1: Anthropic — Multi-Agent Research System
**URL [accessed 2026-05-12]:** https://www.anthropic.com/engineering/built-multi-agent-research-system

"Subagents call tools to store their work in external systems, then pass lightweight references back to the coordinator" — prevents game-of-telephone information loss. **Application:** the pyfinagent lite path violates this by passing a single `reason` string through both Trader and RiskJudge consumers via shared in-memory dict.

### Source 2: Anthropic — Harness Design for Long-Running Apps
**URL [accessed 2026-05-12]:** https://www.anthropic.com/engineering/harness-design-long-running-apps

"Agents tend to respond by confidently praising the work — even when, to a human observer, the quality is obviously mediocre"; "separating the agent doing the work from the agent judging it proves to be a strong lever." **Application:** the RiskJudge reading from `analysis.reason` is the exact "evaluator inheriting generator's reasoning" failure mode.

### Source 3: Anthropic — Building Effective Agents
**URL [accessed 2026-05-12]:** https://www.anthropic.com/engineering/building-effective-agents

"One LLM call generates a response while another provides evaluation and feedback in a loop"; evaluator must be independent. **Application:** the fix is exactly an evaluator-optimizer loop where RiskJudge runs a separate LLM call with a risk-specific prompt.

### Source 4: Du et al. — Society of Minds (LLM Debate)
**URL [accessed 2026-05-12]:** https://composable-models.github.io/llm_debate/

"Agents first propose independently, then critique"; "the final answer generated after such a procedure is both more factually accurate." **Application:** Independent generation BEFORE peer observation is the structural prerequisite. Today's lite path observes Trader's output as the RiskJudge's only input — the opposite of the Society of Minds prescription.

### Source 5: NIST — Building Evaluation Probes into Agentic AI
**URL [accessed 2026-05-12]:** https://www.nist.gov/programs-projects/building-evaluation-probes-agentic-ai

"Automated tools... act as adversarial verifiers" using "a strict rubric" compared to "human-curated corpus of reference documents." **Application:** A risk evaluation grounded only in the trader's reasoning is not adversarial. Real risk-checks compare against an independent reference (e.g., concentration limits, drawdown thresholds) not against the trader's own text.

### Source 6: Anthropic — Demystifying Evals for AI Agents
**URL [accessed 2026-05-12]:** https://www.anthropic.com/engineering/demystifying-evals-for-ai-agents

Distinct grader types (code-based, model-based, human) — "distinct grader types evaluate different aspects without overlap, reducing bias." **Application:** RiskJudge should evaluate concentration / drawdown / governance compliance, which is largely code-based (limits.yaml from bucket 24.1) — not a re-reading of the Trader's narrative.
