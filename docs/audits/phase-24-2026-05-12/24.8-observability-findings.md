# Findings — phase-24.8 — Observability + Safety Rails

## Executive summary

Observability + safety rails in pyfinagent are in better shape than buckets 24.1/24.4/24.5 might suggest, but three critical gaps remain. The kill-switch is operator-reachable: `frontend/src/components/OpsStatusBar.tsx:96-130` exposes PAUSE / RESUME / FLATTEN_ALL buttons that POST to `backend/api/kill_switch_api.py`. Auto-pause on trailing-DD or daily-loss breach correctly fires a Slack alert via `backend/services/kill_switch.py:144-156`. The watchdog at `backend/services/watchdog.py` correctly fires `kickstart -k` + Slack on backend-down — `handoff/logs/backend-watchdog.log` shows 3 successful restarts in 12 days. However: (1) **Cost budget is honor-system only** — `backend/api/cost_budget_api.py` correctly maintains a `tripped=True` flag in BQ, but `backend/services/llm_client.py` does NOT check this flag before LLM API calls; a new cycle after breach still makes Anthropic + Gemini calls normally. (2) **SLA escalation uses non-standard `imsg` binary** with no Slack fallback (P0 alerts go through `imsg` only); if `imsg` is unavailable on the host, the escalation silently fails. (3) **Governance limits-loader watcher** at `backend/governance/limits_loader.py` correctly fires `os._exit(2)` on file digest change (immutability enforcement) but sends no Slack alert before terminating — operator sees only a backend-down event without knowing it was a governance-triggered exit. Five phase-25 candidates close the gaps: cost-budget hard-block in llm_client, SLA Slack fallback, governance pre-exit Slack alert, kill-switch hot-key, observability /status aggregator.

## Code-grounded findings

### F-1: Kill-switch is operator-reachable from `/paper-trading` UI

`frontend/src/components/OpsStatusBar.tsx:96-130`:
- PAUSE button POSTs to `/api/kill-switch/pause`
- RESUME button POSTs to `/api/kill-switch/resume`
- FLATTEN_ALL button POSTs to `/api/kill-switch/flatten`

`backend/api/kill_switch_api.py` routes these to `backend/services/kill_switch.py`. Operator-reachable via the OpsStatusBar component shown on `/paper-trading` page.

### F-2: Auto-pause fires Slack alert (working)

`backend/services/kill_switch.py:144-156`:

```python
def check_and_enforce_kill_switch(self, ...):
    if dd > self.trailing_dd_limit or daily_loss > self.daily_loss_limit:
        self.activate("auto_pause", reason="trailing_DD_breach")
        asyncio.create_task(slack_alert(P0, "Kill Switch Auto-Activated", ...))
```

This is one of the FEW Slack-wired escalation paths in the codebase that works end-to-end. Cross-link bucket 24.5 candidates (which are about expanding this pattern to other notification categories).

### F-3: Watchdog working (verified by log)

`backend/services/watchdog.py` fires:
- `kickstart -k` if backend not responding to health check
- Slack alert "Backend Down — auto-restart attempted"

`handoff/logs/backend-watchdog.log` shows 3 restart events in last 12 days (verified by researcher). Pattern is healthy.

### F-4: Cost budget is honor-system (CRITICAL GAP)

`backend/api/cost_budget_api.py`:
- Tracks cumulative spend per provider (Anthropic, Gemini, etc.)
- Sets `tripped=True` when threshold breached
- Stores state in BQ `cost_budget_state` table

`backend/services/llm_client.py` (the unified LLM call wrapper):
- Has NO check for the `tripped` flag before making API calls
- Calls Anthropic / Gemini regardless of budget state

**Consequence:** if cost budget breaches at 11pm and a new daily cycle starts at 7am the next morning, the cycle runs LLM calls normally, accumulating MORE spend on top of the breach. The budget is observational, not enforcement.

CLAUDE.md says: "LLM API costs require Peder's explicit approval." Today this is enforced only by operator vigilance, not by code. Phase-25 candidate `25.A8` closes this.

### F-5: SLA escalation uses `imsg` only (gap)

`backend/services/sla_monitor.py` (or equivalent):
- P0 alerts route through `imsg` (non-standard macOS-specific binary)
- No Slack fallback
- If `imsg` is unavailable (e.g., on Linux deployment, on macOS without iMessage configured), P0 escalations silently fail

Note: `backend/slack_bot/scheduler.py:369-423` `send_trading_escalation` HAS the infrastructure for both iMessage and Slack (per bucket 24.5 F-7). The gap is wiring: SLA monitor should call `send_trading_escalation` rather than a direct `imsg` binary call.

### F-6: Governance watcher exits without Slack alert

`backend/governance/limits_loader.py:156` (immutability watcher):
- Computes file digest at boot
- Watches for digest change
- On change: `os._exit(2)` — process termination

**Gap:** no Slack alert is sent BEFORE `os._exit(2)`. The watchdog (F-3) will detect backend-down and restart it, sending a generic "Backend Down" alert. Operator does not see "Governance limits.yaml was tampered — backend killed by limits watcher." They see a generic restart event.

Cross-link bucket 24.1 — governance limits are immutable but unused in trade path. So today: the watcher fires only if someone edits limits.yaml (rare); but when it fires, the operator gets uninformative signal.

### F-7: Observability API surface

`backend/api/observability_api.py`:
- `/observability/freshness` (delegates to cycle_health — bucket 24.7 gap)
- Limited additional endpoints

There's no `/observability/status` aggregator that returns kill-switch state + watchdog last-check + cost-budget remaining + governance digest + SLA breach count in one call. Frontend `OpsStatusBar.tsx` would benefit from such a single endpoint.
