# Findings — phase-24.3 — Autoresearch ↔ Daily-Loop Wiring

## Executive summary

The autoresearch and meta_evolution subsystems exist as production-grade libraries with weekly + monthly schedules, but they are entirely decoupled from `backend/services/autonomous_loop.py` — the daily trading cycle has zero imports from either subsystem (verbatim grep evidence). Promoted strategies are written to a flat TSV ledger at `backend/autoresearch/weekly_ledger.tsv` with no subscriber. `backend/autoresearch/monthly_champion_challenger.py:76` hard-codes `actual_replacement: False`, so even HITL-approved monthly challengers produce no downstream change to live trading. `backend/autoresearch/cron.py:29-38` registers a stub `func=lambda: None` — the nightly autoresearch cron does NOT run experiments on schedule (APScheduler wiring "deferred to phase-9" per inline comment). The Sunday weekly `meta_evolution/cron.py:87-152` only touches YAML budget allocation (cron_allocator + provider_rebalancer + archetype_library) — it never reads promoted IDs and never writes to `optimizer_best.json`. The daily cycle reads `optimizer_best.json` at startup (`autonomous_loop.py:33-43`), which is in the `Proposer` whitelist — but no scheduled path writes to it. The broken edge is: autoresearch outputs → [no listener] → autonomous_loop. Three phase-25 candidates: (a) extend `friday_promotion.py` to write a `pyfinagent_data.promoted_strategies` BQ table; (b) add `load_promoted_params()` to `autonomous_loop.py:99-107`; (c) implement an industry-canonical `strategy_registry` BQ table with `status` field (active/shadow/retired) and flip `actual_replacement` from `False` to gated-on-registry-write. Plus two safety candidates (shadow-mode A/B and rollback wiring).

## Code-grounded findings

### F-1: Verbatim grep — zero cross-module imports

```
$ grep -rn "autoresearch\|meta_evolution" backend/services/autonomous_loop.py
(no output)
```

Confirmed. The daily trading cycle has no awareness of either subsystem.

### F-2: meta_evolution Sunday cron is YAML-only

`backend/meta_evolution/cron.py:87-152` (`run_meta_evolution_cycle`):

- Calls `cron_allocator.allocate` (YAML budget writes)
- Calls `provider_rebalancer.allocate` (YAML provider budget writes)
- Calls `archetype_library` count (read-only)
- Computes `alpha_velocity` → writes to BQ `pyfinagent_pms.alpha_velocity_samples` (never read back)
- Calls `directive_rewriter` + `directive_review` (LLM prompt mutations to `researcher.md`; HITL-only)

**Does NOT:** read `weekly_ledger.tsv`, read `fri_promoted_ids`, write to `optimizer_best.json`, query promoted-strategies registry, or notify the daily loop.

### F-3: Friday promotion writes to flat TSV with no listener

`backend/autoresearch/friday_promotion.py:121-131`:

```python
weekly_ledger.append_row({
    "week_iso": ...,
    "fri_promoted_ids": ...,
    "dsr": ..., "pbo": ...
})
```

`LEDGER_PATH = backend/autoresearch/weekly_ledger.tsv` (from `weekly_ledger.py`). This is a local file. No process reads it. The daily loop, the harness, the Slack bot, and the frontend all ignore it.

### F-4: Monthly champion/challenger hard-codes no-op

`backend/autoresearch/monthly_champion_challenger.py:76`:

```python
# Hard invariant: actual_replacement is always False.
actual_replacement: bool = False
```

Even when the HITL 48h window approves a challenger via `record_approval`, `actual_replacement` stays `False`. The gate logic (Sortino delta + DSR threshold) runs correctly; its outputs produce no downstream action by design.

### F-5: Autoresearch nightly cron is a stub

`backend/autoresearch/cron.py:29-38`:

```python
def register(scheduler):
    scheduler.add_job(
        func=lambda: None,  # APScheduler wiring deferred to phase-9
        trigger=...,
        id="autoresearch_nightly",
    )
```

The lambda is the stub. Zero experiments run on schedule. The Thursday batch sampler (`thursday_batch.py`), Friday promoter (`friday_promotion.py`), gate (`gate.py`), promoter (`promoter.py`), and rollback (`rollback.py`) are all production-grade libraries with no scheduled caller.

### F-6: `autonomous_loop.py` reads `optimizer_best.json` only

`backend/services/autonomous_loop.py:33-43` calls `load_best_params()` at cycle start. This reads `backend/backtest/experiments/optimizer_best.json`. The file is in `Proposer` whitelist (`backend/autoresearch/proposer.py:24-26`) — the proposer CAN write to it — but no scheduled trigger invokes the proposer. So `optimizer_best.json` only changes when an operator or harness cycle explicitly proposes new params. Cross-link bucket 24.6 (backtest engine) — the harness IS one such writer, but it operates on optimization, not on autoresearch promotion.

### F-7: Slot accounting writes UI-only

`backend/autoresearch/slot_accounting.py:26` writes to BQ `pyfinagent_data.harness_learning_log`. Read path: `backend/api/harness_autoresearch.py` (`/api/harness/sprint-state` endpoint) → frontend Harness tab. The daily loop does NOT query this table. So even rich BQ-stored autoresearch state is not load-bearing for trade decisions.

### F-8: Industry consensus — registry-alias polling

External research (Snowflake, DataRobot, Databricks) converges on the "model registry with status alias" pattern. The daily serving loop polls `WHERE status='active' ORDER BY promoted_at DESC LIMIT 1`. Promotions update the alias atomically. pyfinagent has all the upstream pieces (proposer, gate, promoter, rollback) but the registry table and the poll-on-cycle-start are missing.
