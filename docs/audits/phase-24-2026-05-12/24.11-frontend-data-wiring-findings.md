# Findings — phase-24.11 — Frontend↔Backend Wiring Data Layer

## Executive summary

Frontend↔backend wiring is mostly clean. 119 backend routes are registered in `backend/main.py:379-413`, and `frontend/src/lib/api.ts` exposes 83 typed API functions covering the operator-visible surface. The hypothesis is confirmed: the `/paper-trading/learnings` page is an **orphan** — `frontend/src/app/paper-trading/learnings/page.tsx:6-9` includes an inline comment "Live data hookup lands in a follow-up backend step" and renders `VirtualFundLearnings` with no data; the component (`VirtualFundLearnings.tsx:285 LOC`) only renders empty states. No `/api/paper-trading/learnings` route exists; `backend/api/paper_trading.py` has 23 routes (none named learnings). Type drift between Pydantic models and TypeScript interfaces is minimal but exists at two points: `ReportSummary.analysis_date` is `datetime` in Pydantic (`backend/api/models.py:96`) but `string` in TS (`frontend/src/lib/types.ts:120`) — benign because Python JSON serializer emits ISO strings; and `SynthesisReport` enrichment fields use `Optional[dict]` in Pydantic but fully-typed sub-interfaces in TS — TS is more precise, meaning naive codegen-from-Pydantic would REGRESS these types. Seven `unknown`/`Record<string, unknown>` return types in `api.ts` defeat TypeScript safety for `getReport`, `getSignal`, `getMacroIndicators`, `getPaperCyclesHistory`, `getPaperLivePrices`, `runBacktest`, `runDataIngestion`. The auth header (`Authorization: Bearer ${token}`) is correctly propagated via `apiFetch` middleware. `GoLiveGate` type is exported from a component file (anti-pattern) and sovereign interfaces are inline in `api.ts:568-709` instead of `types.ts`. Four phase-25 candidates: (A) wire learnings backend, (B) OpenAPI-based TS codegen for drift prevention, (C) consolidate stray types into types.ts, (D) replace `unknown` return types with concrete interfaces.

### F-2: Type drift at 2 points (minimal)

**Drift point 1 — `ReportSummary.analysis_date`:**
- Pydantic: `analysis_date: datetime` (`backend/api/models.py:96`)
- TS: `analysis_date: string` (`frontend/src/lib/types.ts:120`)

Benign — Python JSON serializer emits ISO 8601 strings; TS receives strings. Unintentional but harmless. Any consumer that needs Date objects parses with `new Date(s)`.

**Drift point 2 — `SynthesisReport` enrichment fields:**
- Pydantic: `enrichment_signals: Optional[dict]`, `debate_result: Optional[dict]`, `risk_data: Optional[dict]`, `bias_report: Optional[dict]`, `conflict_report: Optional[dict]` (`backend/api/models.py:52-56`)
- TS: fully-typed sub-interfaces for each (`frontend/src/lib/types.ts:67-89`)

The TS types are MORE precise than the Pydantic source. **Implication:** naive codegen-from-Pydantic would regress these to `Record<string, unknown>` — operator would lose type safety. Any codegen approach must allow override files where TS types are more precise.

### F-3: 7 `unknown` return types in api.ts

`frontend/src/lib/api.ts`:
- `getReport(id): Promise<unknown>` (~line 100)
- `getSignal(id): Promise<unknown>` (~line 150)
- `getMacroIndicators(): Promise<Record<string, unknown>>` (~line 180)
- `getPaperCyclesHistory(): Promise<Record<string, unknown>[]>` (~line 250)
- `getPaperLivePrices(): Promise<Record<string, unknown>>` (~line 280)
- `runBacktest(...): Promise<unknown>` (~line 400)
- `runDataIngestion(...): Promise<unknown>` (~line 440)

Each is a typed-API escape hatch. Components consuming these functions cast or use `any` downstream, defeating compile-time type checks. Phase-25 candidate D replaces with concrete interfaces.
