# Findings — phase-24.1 — Trading-Execution + Governance Audit

## Executive summary

The most user-impactful live bug in pyfinagent is confirmed: `check_stop_losses()` at `backend/services/paper_trader.py:414-423` is orphan code — defined once, called from zero production paths. The autonomous daily loop wires `check_and_enforce_kill_switch()` at `autonomous_loop.py:314` (NAV-level protection works) but has no equivalent per-position stop-enforcement step. As a result 6 of 11 current positions (ON, INTC, TER, DELL, GLW, CIEN) carry `stop_loss_price=None`, and TER has accumulated -12.30% unrealized P&L with zero sell action because `portfolio_manager.py:82-88` silently bypasses the stop check when `stop_loss_price` is None (Python truthiness on `if stop and current ...`). A second-order governance gap compounds the problem: `backend/governance/limits.yaml:28` defines `max_sector_weight_pct: 0.30` (immutable) but the value is never consulted by `decide_trades()`, which instead uses an unrelated count-based cap `paper_max_per_sector` defaulting to 0 (disabled). Five P0/P1 phase-25 candidates close the gap: wire stop-enforcement step, backfill missing stops with same-cycle re-check, no-sells-watchdog, sector-weight enforcement from immutable limits, and notional-cap enforcement in `execute_buy()`. A sixth P0 candidate (no-stop-on-entry hard block) prevents future regressions.

## Code-grounded findings

### F-1: `check_stop_losses()` is orphan code (paper_trader.py:414-423)


Defined once at `backend/services/paper_trader.py:414`. The four other matches are in the master audit prompt (this audit's own prompt referencing the bug). Zero production callers exist anywhere in `backend/`, `scripts/`, or `tests/`. **Hypothesis confirmed.**

The function body iterates current positions, compares `current_price` to `stop_loss_price`, and returns the list of tickers at or below their stop. It does NOT execute sells — the caller would need to invoke `execute_sell(ticker, reason="stop_loss_trigger")` for each returned ticker. Without a caller, the function does nothing.

### F-2: Autonomous loop never calls `check_stop_losses()` (autonomous_loop.py — full file scan)

`backend/services/autonomous_loop.py` is 904 lines. A direct full-file grep for `check_stop_losses` returns zero matches. The closest scaffolding is at `autonomous_loop.py:314`:

```python
trader.check_and_enforce_kill_switch()
```

This call enforces NAV-level kill-switch (4% daily-loss, 10% trailing drawdown) but does not invoke per-position stop checks. Step 5.5 of the cycle (the position-management step) flows directly from `mark_to_market` (Step 5) → `check_and_enforce_kill_switch` (Step 5.5) → `decide_trades` (Step 6). There is no Step 5.6 for stop-loss enforcement.

### F-3: `decide_trades()` stop check silently bypasses None stops (portfolio_manager.py:82-88)

```python
# portfolio_manager.py:82-88
stop = pos.get("stop_loss_price")
current = pos.get("current_price", 0)
if stop and current and current <= stop:
    orders.append(TradeOrder(ticker=ticker, action="SELL", reason="stop_loss", ...))
    continue
```

Python truthiness: `if stop and current...` evaluates to `False` when `stop is None` (regardless of `current`), short-circuiting the check. The 6 positions with `stop_loss_price=None` are never flagged for SELL. Furthermore this check only runs within `decide_trades()`, which is called once per autonomous cycle. Between cycles (intraday), no stop check fires at all.

### F-4: `_extract_stop_loss()` fallback only applies to NEW buys (portfolio_manager.py:288-329)

The phase-23.1.8 fallback at `portfolio_manager.py:322-328` resolves stop in three steps:
1. `risk_assessment.risk_limits.stop_loss` (absolute price from LLM)
2. `risk_assessment.risk_limits.stop_loss_pct` (% below entry from LLM)
3. `settings.paper_default_stop_loss_pct` (8% default fallback)

But the fallback requires `price` (i.e., `analysis.get("price_at_analysis")`) to be non-None. The fallback fires ONLY at trade-entry time. Existing positions bought before phase-23.1.8 (which added this fallback) were never backfilled. The 6 stop-less positions are approximately 15 days old (per operator), pre-dating the phase-23.1.8 deployment.

### F-6: TER -12.30% no-sell case — root cause trace

TER is the most acute failure: -12.30% unrealized P&L with no stop and no sell.

**Why no sell occurred:**
1. `check_stop_losses()` at `paper_trader.py:414` — orphan; never runs
2. `portfolio_manager.py:82` stop check — TER has `stop_loss_price=None`, so the `if stop and current ...` short-circuits to False; check silently bypassed
3. `decide_trades()` SELL logic (`portfolio_manager.py:92-113`) only triggers SELL on:
   - LLM SELL/STRONG_SELL recommendation (TER may still get HOLD from re-evaluation)
   - HOLD-to-something downgrade
4. Kill-switch (`autonomous_loop.py:314`) is NAV-level; TER's -12.30% position loss is NOT large enough to breach the 4% daily-loss or 10% trailing-drawdown thresholds on a portfolio with 10+ positions

Sell-caller inventory:
```

backend/services/portfolio_manager.py:86:  TradeOrder(... action="SELL" ...)  # stop logic (bypassed when stop_loss_price is None)

```

The autonomous loop at `autonomous_loop.py:396-409` calls `trader.execute_sell(...)` for SELL orders emitted by `decide_trades()`. Since `decide_trades()` does not emit SELL for TER (no-stop + LLM HOLD), TER has never been sold.

### F-7: Governance gap — limits_loader watcher vs decide_trades

`backend/governance/limits.yaml:28` defines:
```yaml
max_sector_weight_pct: 0.30
max_position_notional_pct: 0.05
```

These are **immutable** (watcher kills process on file digest change, SIGHUP ignored). The watcher at `backend/governance/limits_loader.py:156` works.

But the trade decision layer at `backend/services/portfolio_manager.py:194` uses:
```python
max_per_sector = int(getattr(settings, "paper_max_per_sector", 0) or 0)
```

This is a **count-based** cap from `Settings` (defaults to 0 = disabled). It is NOT the immutable weight-based cap from `limits.yaml`. The 30% sector-weight cap is never consulted in any trade path — `grep -rn "max_sector_weight_pct" backend/` returns only `governance/limits_schema.py` and `governance/limits.yaml`, with zero callers in `services/`.

Similarly, `max_position_notional_pct: 0.05` (5% max per symbol) is never enforced in `execute_buy()` at `paper_trader.py:97-99`, which only checks `len(positions) >= paper_max_positions` (count cap on number of positions).

**Architectural conclusion:** the immutable limits exist with watchdog enforcement, but the trading code does not load them. The limits.yaml limits-loader watcher works for *configuration tamper detection* but the limits themselves are not load-bearing for trading decisions. This is a governance gap.

## External-research summary

### Source 1: Anthropic — Harness Design for Long-Running Application Development
**URL [accessed 2026-05-12]:** https://www.anthropic.com/engineering/harness-design-long-running-apps

Verbatim quotes informing this bucket:

- "Agents tend to respond by confidently praising the work — even when, to a human observer, the quality is obviously mediocre."

**Application:** `check_stop_losses()` is exactly the "stale scaffolding" failure mode — code written under one set of assumptions (intent to wire) that became orphan when the wiring was never completed. The "agents praise own work" warning explains why prior cycles did not catch it: no external evaluator was checking that the function had callers. Phase-24 is the external evaluator.

### Source 2: arXiv 2604.27150 — Optimal Stop-Loss and Take-Profit Parameterization for Autonomous Trading Agent Swarm
**URL [accessed 2026-05-12]:** https://arxiv.org/html/2604.27150

Empirical result: "a 10% stop-loss, shortening the trailing distance, and taking partial profit earlier improves both Sharpe and profit factor" vs a 25% baseline. ATR 1.0x overlay raises Sharpe 56% above baseline.

**Application:** pyfinagent's `paper_default_stop_loss_pct = 8.0` is in the right empirical range (O'Neil canonical 8%, arxiv recommends 10%). The bug is NOT in the stop *value* but in stop *enforcement*. After Candidate 25.1 wires basic enforcement, a future enhancement (post phase-24 audit) is to make stops ATR-adaptive.

### Source 3: Frontiers in Behavioral Economics 2024 — Debiasing the Disposition Effect
**URL [accessed 2026-05-12]:** https://www.frontiersin.org/journals/behavioral-economics/articles/10.3389/frbhe.2024.1345875/full

Fischbacher et al. (2017) showed: stop-loss/take-gain orders mitigate the disposition effect *only when binding* (automated, force-executed). Reminders, warnings, and advisory notes are ineffective.

**Application:** The 6 stop-less pyfinagent positions held in the red are a textbook disposition-effect scenario. `check_stop_losses()` returning a list (a kind of reminder) without auto-executing sells is the "advisory" failure mode the literature warns against. Candidate 25.1 must wire BOTH halves — detection AND auto-sell.

### Source 4: Alpaca — Orders at Alpaca
**URL [accessed 2026-05-12]:** https://docs.alpaca.markets/docs/orders-at-alpaca

17 order states: new, partially_filled, filled, done_for_day, canceled, expired, replaced, pending_cancel, pending_replace, accepted, pending_new, accepted_for_bidding, stopped, rejected, suspended, calculated. Stop price must be $0.01 below base price to prevent race conditions.

**Application:** Alpaca paper trading does NOT auto-execute stops unless an explicit stop order is submitted. pyfinagent persists `stop_loss_price` in BQ but never submits a stop order to Alpaca. Two parallel enforcement paths:
- BQ-sim path: `check_stop_losses()` must be wired into the cycle to synthesize stop execution
- Alpaca path: `execute_buy()` must also submit a `stop_loss` child order alongside the market buy

### Source 5: Semnet — Agentic AI Governance in 2026
**URL [accessed 2026-05-12]:** https://www.semnet.co/post/agentic-ai-governance-in-2026-preventing-data-leaks-and-cves

Kill-switch controller: "disables tools, models, or whole agents in one call and coordinates rollback." Agent drift after model/prompt updates can trigger unauthorized write actions.

**Application:** pyfinagent's kill_switch.py is principle-aligned (NAV-level kill via `check_and_enforce_kill_switch()`) but lacks per-position stop enforcement. The right pattern: NAV-level kill-switch protects against tail events; per-position stops are the first line of defense. Both must be live.
