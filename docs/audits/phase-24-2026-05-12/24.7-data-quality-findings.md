# Findings — phase-24.7 — Data Quality + BQ Freshness + yfinance Fallback

## Executive summary

The `/freshness` endpoint at `backend/services/cycle_health.py:214-228` covers only `paper_trades` + `paper_portfolio_snapshots`, leaving five critical tables (`historical_prices`, `historical_fundamentals`, `historical_macro`, `signals_log`, plus paper-trading positions) unmonitored. A surprise routing bug surfaced: `backend/backtest/data_ingestion.py:34` sets the historical-table dataset via `settings.bq_dataset_reports = "financial_reports"` — the `historical_*` tables actually live in the `financial_reports` dataset, NOT in `pyfinagent_hdw` as CLAUDE.md's BQ dataset table describes. This is either stale documentation or an unfinished migration; either way, BQ MCP queries against `pyfinagent_hdw` for these tables return nothing. The yfinance fallback path fires silently: `backend/agents/orchestrator.py:1141` logs at INFO level (suppressed at the default WARNING tier), no counter is persisted, and no Slack alarm fires when fallback dominates. `backend/tools/yfinance_tool.py:84-88` `get_price_history()` has zero exception handling and zero logging — bare unguarded `stock.history()` call. `backend/backtest/cache.py:184-228` `preload_macro()` has an idempotency guard but no max-age check, so stale macro silently powers full backtest runs. Three phase-25 candidates (with two more from the cross-link bucket): (A) per-table freshness endpoint covering all five data tables with SLA bands, (B) yfinance-fallback-fired counter persisted to BQ + log promotion to WARNING, (C) unified `/api/observability/data-freshness` endpoint distinct from paper-trading freshness; plus (D) `preload_macro()` max-age guard, (E) `yfinance_tool.get_price_history()` try/except + counter.

## Code-grounded findings

### F-1: `/freshness` endpoint blind to historical tables

`backend/services/cycle_health.py:214-228` queries only:
- `paper_trades` (recent trade actions)
- `paper_portfolio_snapshots` (NAV snapshots)

Tables NOT queried:
- `historical_prices` (yfinance ingest target)
- `historical_fundamentals` (yfinance ingest target)
- `historical_macro` (FRED + macro feed)
- `signals_log` (every analysis cycle writes a row at `bigquery_client.py:386-392`)

The endpoint name implies broad "freshness" coverage but actual coverage is narrow.

### F-2: Dataset routing surprise — historical tables in `financial_reports`, not `pyfinagent_hdw`

`backend/backtest/data_ingestion.py:34`:

```python
self.dataset = settings.bq_dataset_reports  # = "financial_reports"
```

`settings.py:40` defines `bq_dataset_reports = "financial_reports"`. So the three historical tables are written to `financial_reports.historical_*`, not `pyfinagent_hdw.historical_*`. CLAUDE.md's BQ dataset table describes `pyfinagent_hdw` as "Historical data warehouse" — stale doc OR unfinished migration. The audit prompt at `scripts/audit/phase_24_audit_prompt.md` for bucket 24.7 also mentions inspection via `mcp__bigquery__describe-table` on `pyfinagent_data.signals` — but `signals_log` is one of the active write targets, and `historical_*` are in `financial_reports`. Cross-link bucket 24.10 (MCP) for `describe-table` smoke test.

### F-3: yfinance silent fallback at `orchestrator.py:1141`

```python
# backend/agents/orchestrator.py:1141
logger.info(f"yfinance fallback fired for ticker {ticker}")
```

CLAUDE.md: "use WARNING for quiet terminals." Default log level WARNING suppresses INFO. No counter, no Slack alarm, no BQ persistence. Operator has zero visibility when yfinance is the primary data source.

### F-4: `yfinance_tool.get_price_history()` is unguarded

`backend/tools/yfinance_tool.py:84-88`:

```python
def get_price_history(self, ticker, period="1mo"):
    stock = yf.Ticker(ticker)
    return stock.history(period=period)
```

No try/except. No logging. No retry. If Yahoo rate-limits or returns empty DataFrame, the caller propagates the failure (or, worse, silently uses empty data). Callers include `monte_carlo.py`, `anomaly_detector.py`. Compare with `get_comprehensive_financials()` which catches exceptions and returns `{"error": ...}` dict — the price function does not have that safety.

### F-5: `preload_macro()` has idempotency but not freshness guard

`backend/backtest/cache.py:184-228`:

```python
def preload_macro(self):
    if self._macro_preloaded:
        return  # already preloaded, skipping BQ query
    # ... BQ query ...
    self._macro_preloaded = True
```

The idempotency at L193 prevents re-querying within the same process, but if `historical_macro` was last updated 3 weeks ago, the cached data is 3 weeks stale and the entire backtest run uses it silently. CLAUDE.md says: "Always call `cache.preload_macro()` or backtests hang after ~40min." So this is hot-path code — adding a max-age check is critical.

### F-6: `data_ingestion.py:113` handles `data.empty` but not per-ticker empties

```python
if data is None or data.empty:
    continue
```

This protects against TOTAL batch failure. But `yf.download(tickers=[multi], group_by="ticker")` can return a DataFrame where individual tickers have all-NaN columns — `data.empty` is False (other tickers have data) and the per-ticker partial empty slips through. The per-ticker `try/except` at L155 catches only processing exceptions, not data-quality issues.

### F-7: `signals_log` writes invisible to freshness

`backend/db/bigquery_client.py:386-392` writes every analysis cycle's signals to `pyfinagent_data.signals_log`. But `cycle_health.py` does NOT query this table. The most direct measure of "are signals being produced" is unmonitored.

## External-research summary

### Source 1: Metaplane — BQ Freshness Methods (full)
**URL:** https://www.metaplane.dev/blog/stay-fresh-four-ways-to-track-update-times-for-bigquery-tables-and-views
Four methods: `MAX(ts)`, `__TABLES__` metadata, `INFORMATION_SCHEMA`, `_PARTITIONTIME`. Critical: metadata `last_change_time` tracks SCHEMA changes, not DATA writes. **Application:** `cycle_health.py` correctly uses `MAX(event_timestamp)` — sound method.

### Source 2: OneUptime — Dataplex Data Quality (Feb 2026, full)
**URL:** https://oneuptime.com/blog/post/2026-02-17-how-to-build-a-data-quality-monitoring-framework-on-bigquery-using-dataplex-data-quality-tasks/view
Timeliness: `TIMESTAMP_DIFF(CURRENT_TIMESTAMP(), _processed_at, HOUR) < 2`; Slack via Cloud Functions; 95% pass_rate. **Application:** Candidate A SLA bands.

### Source 3: Manik Hossain — Freshness Monitoring (2026, full)
**URL:** https://medium.com/@manik.ruet08/monitoring-data-freshness-across-large-analytics-platforms-5632fd0e5722
Three timestamp types (event/ingestion/load); `dashboard_age = MIN(upstream_table_ages)`. **Application:** for `/freshness` to be honest, it must aggregate the MIN across all 5 tables, not report 1 table.

### Source 4: OneUptime — Python Circuit Breakers (Jan 2026, full)
**URL:** https://oneuptime.com/blog/post/2026-01-23-python-circuit-breakers/view
PyBreaker with fallback_function; Closed/Open/Half-Open states. **Application:** yfinance fallback should expose state metrics — Candidate B counter is the minimal viable instrument.

### Source 5: Craakash — yfinance Pitfalls (2025, full)
**URL:** https://medium.com/@craakash/how-to-get-stock-data-without-yfinance-code-examples-2025-edition-data-pipeline-b51f1ecc906d
yfinance scrapes; no SLA; silently returns empty DataFrames. **Application:** every yfinance call site needs explicit empty-data check + counter.

### Source 6: GCP — BigQuery Best Practices Input (full)
**URL:** https://cloud.google.com/bigquery/docs/best-practices-performance-input
`_PARTITIONTIME` for ingestion-partitioned tables; INFORMATION_SCHEMA.JOBS for execution. **Application:** for per-partition freshness (Manik pattern).
