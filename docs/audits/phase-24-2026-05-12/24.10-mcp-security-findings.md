# Findings — phase-24.10 — MCP Infrastructure + Security

## Executive summary

pyfinagent's MCP setup is principle-aligned with 2026 best practices: two pinned servers (`alpaca-mcp-server==2.0.1`, `mcp-server-bigquery==0.3.2`) at `.mcp.json`, deny-listed write paths at `.claude/settings.json:153-158`, stdio transport only (no HTTP MCP proxy, so no confused-deputy / SSRF / token-passthrough risk), and a complete secrets rotation schedule at `scripts/ops/secrets_rotation_schedule.json` with no overdue entries at 2026-05-12. NextAuth v5 with WebAuthn (passkey) is correctly wired at `frontend/src/lib/auth.ts:15` with httpOnly+Secure session cookies via middleware. Three security gaps to close: (1) **Alpaca tool inventory not exhaustively audited** — `.claude/settings.json:153-158` denies 5 known Alpaca write tools, but no smoke test enumerates the full server tool surface; a minor version bump could silently add unblocked write tools. (2) **Plain `str` API keys in `backend/config/settings.py:87-92,192`** — `anthropic_api_key`, `openai_api_key`, `alpaca_api_secret_key`, `auth_secret`, `slack_bot_token` should be `SecretStr` for log-masking. (3) **No Alpaca MCP smoke test** — `scripts/mcp_servers/` has only `smoke_test_bigquery_mcp.py`. Spot-check of git history confirms no committed secrets; backend/.env is correctly gitignored. Tool poisoning is the dominant client-side risk per arXiv 2603.22489; pyfinagent's version-pin discipline + deny-list mitigates but the full-surface audit is missing. Six phase-25 candidates proposed: Alpaca tool-surface smoke test, SecretStr migration, Finnhub MCP, Alpha Vantage MCP, Alpaca API key rotation script wiring, MCP supply-chain digest pinning.

## Code-grounded findings

### F-1: `.mcp.json` is correctly pinned

`.mcp.json` lines 1-25:

```json
{
  "mcpServers": {
    "alpaca": {
      "command": "uvx",
      "args": ["alpaca-mcp-server==2.0.1"],
      ...
    },
    "bigquery": {
      "command": "uvx",
      "args": ["mcp-server-bigquery==0.3.2", "--project", "...", "--location", "US"],
      ...
    }
  }
}
```

Both servers pinned to exact versions. Supply-chain risk minimal — `uvx` resolves from PyPI with reproducible hash. Per arxiv 2603.22489 + Red Hat MCP security guide, version pinning is the load-bearing mitigation against tool-poisoning via upstream update.

### F-2: Deny rule surface — `.claude/settings.json:153-158`

```json
"deny": [
  "Bash(curl:*)",
  "mcp__alpaca__create_order",
  "mcp__alpaca__cancel_all_orders",
  "mcp__alpaca__close_position",
  "mcp__alpaca__close_all_positions",
  "mcp__bigquery__execute-query"  // <-- write-class queries gated
]
```

5 Alpaca write tools denied; `execute-query` on BigQuery is denied (read-only ops via `list-tables` + `describe-table`). **Gap:** the deny list was authored against a known Alpaca tool inventory at adoption time; phase-23.2.21 pinned `alpaca-mcp-server==2.0.1`. If a future minor version (2.1.0, 2.2.0) adds new write tools (e.g., `withdraw_funds`, `transfer_to_external`), they would be silently allowed because there's no smoke test that enumerates the live tool surface and reconciles against the deny list.

### F-3: NextAuth v5 + WebAuthn correctly wired

`frontend/src/lib/auth.ts:15`:

```typescript
experimental: { enableWebAuthn: true }
```

`frontend/src/middleware.ts:1-37` enforces session-cookie auth on protected routes. Production passkey-security per the 2025 security analysis: ✓ HTTPS enforced (Next.js production default), ✓ httpOnly + Secure cookies, ✓ session expiry via NextAuth config, ✓ rate limiting via NextAuth v5 default. **Caveat:** `experimental` flag means the API may change with the next major NextAuth bump — `enableWebAuthn` is provisional until NextAuth promotes it to stable.

### F-4: `backend/.env` secret handling — no committed secrets

Spot-checks:
- `git log --all --diff-filter=A -- backend/.env` — empty result (never committed)
- `git grep "sk-ant-"` — no Anthropic key in any tracked file
- `git grep "AKIA"` — no AWS key
- `backend/.env.example` exists with placeholders only

`.gitignore` correctly excludes `backend/.env` and `*.env`. Per pydantic-settings best practice (2026 source), `backend/config/settings.py:36` correctly uses `model_config = SettingsConfigDict(env_file=".env", env_file_encoding="utf-8", extra="ignore")`.

**Gap:** `backend/config/settings.py:87-92` defines API keys as plain `str`:

```python
anthropic_api_key: str = ""
openai_api_key: str = ""
alpaca_api_secret_key: str = ""  # line 192
auth_secret: str = ""
slack_bot_token: str = ""
```

These should be `SecretStr` (pydantic.SecretStr) so they mask in logs and `repr()` output. Today a stack trace including `repr(settings)` could leak keys.

### F-5: `secrets_rotation_schedule.json` is current

`scripts/ops/secrets_rotation_schedule.json` lists 11 secrets with `last_rotated` and `next_rotation` timestamps. Spot-check at 2026-05-12: no entries past `next_rotation`. `scripts/ops/secrets_rotation_check.py` runs the audit — works correctly but its output (the audit results) is never read by any downstream process (no Slack alert, no scheduled cron). Operator must run it manually.

### F-6: No Alpaca MCP smoke test

`scripts/mcp_servers/` contains:
- `smoke_test_bigquery_mcp.py` (157 lines, correct)

No equivalent for Alpaca. Per the audit-prompt scope-3 reminder (`backend/CLAUDE.md::BigQuery Access (MCP)`), every adopted MCP should have a versioned smoke test that exercises the live tools and confirms deny rules. Alpaca has been adopted since phase-17 but the smoke test was not added.

### F-7: NextAuth WebAuthn — flow trace

`backend/auth/` (full):
- Auth handled at the NextAuth v5 level — backend has no auth code
- The `/api/auth/*` routes are NextAuth-managed

WebAuthn flow:
1. Frontend `/auth/passkey` → calls NextAuth API
2. NextAuth's WebAuthn provider issues attestation challenge
3. Browser navigator.credentials.create() prompts user (Touch ID / Yubikey)
4. Server verifies signature, stores credential ID in `auth_credentials` table
5. Future logins: navigator.credentials.get() → verify against stored credential

Pattern is correct. No backend auth code means the security surface is the NextAuth library + browser WebAuthn API, both well-audited.

### F-8: MCP tool poisoning mitigation status

Per arXiv 2603.22489: tool poisoning is the most prevalent client-side MCP attack. 5/7 MCP clients lack static validation of tool metadata. pyfinagent uses Claude Code as MCP client — Claude Code does validate tool surfaces against `.claude/settings.json` permissions but does NOT do static schema validation against a known-good baseline. So if a poisoned `alpaca` server returned a tool with a misleading description ("read_only_inspect_orders" but actually placing trades), Claude Code would route based on the description.

**Mitigation in place:** version pinning + deny-list. Each protects against different vectors. Together they make poisoning a high-bar attack — requires either (a) the pinned PyPI package being compromised + the SHA256 not validated, or (b) the deny-list missing the poisoned tool name.
