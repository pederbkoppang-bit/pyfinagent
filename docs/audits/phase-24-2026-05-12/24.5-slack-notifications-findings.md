# Findings — phase-24.5 — Slack Notifications + Operator Alerting

## Executive summary

All four operator-reported Slack bugs are confirmed and rooted to specific file:line anchors in `backend/slack_bot/`. The `Portfolio: +$0.00 (+0.0%)` digest output is a TWO-LEVEL bug: `scheduler.py:235` calls `/api/portfolio/performance` (the legacy in-memory portfolio at `backend/api/portfolio.py` whose `_positions: dict` is always empty after restart) AND `formatters.py:322` reads `total_return_pct` while the endpoint returns `total_pnl_pct`. The "Recent Analyses 5x SNDK" bug is `bigquery_client.py:258-268` running `ORDER BY analysis_date DESC LIMIT 5` with zero ticker dedup — if SNDK was analyzed 5 times recently, all 5 rows are SNDK. The mis-scheduled "2 PM morning digest" is NOT a code bug — `scheduler.py:144` correctly uses `ZoneInfo("America/New_York")` — the operator has wrong values in `.env` (`MORNING_DIGEST_HOUR=14` should be `6`). The missing notification categories (trade confirmations, kill-switch alerts, drawdown alarms, cost-budget breach alerts, cycle-completion summaries, error escalation routing, weekly autoresearch summary) are unimplemented despite the infrastructure (`send_trading_escalation` at `scheduler.py:369-423` + `format_escalation_alert` at `formatters.py:624-686`) being production-ready — zero call sites wire these triggers. `/portfolio` slash command at `commands.py:138` reproduces the same wrong-endpoint bug. Ten phase-25 candidates close the gaps.

## Code-grounded findings

### F-1: Wrong P&L data source (Portfolio: +$0.00 root cause)

`backend/slack_bot/scheduler.py:235` (morning digest) and `scheduler.py:260` (evening digest) call:

```python
GET /api/portfolio/performance
```

This routes to `backend/api/portfolio.py:23` which uses `_positions: dict` — an in-process dict that is ALWAYS EMPTY on a fresh backend restart. Its response (`portfolio.py:99-107`) returns `total_pnl: 0`, `total_pnl_pct: 0` whenever the dict is empty. The digest receives `+$0.00 (+0.0%)`.

The CORRECT live portfolio P&L is at `/api/paper-trading/portfolio` which routes to `backend/api/paper_trading.py:175-208` and reads from `paper_trader.get_positions()` (BQ-backed, 11 positions). Operator-reported "wrong P&L" matches exactly: digests hit the wrong endpoint.

### F-2: `total_return_pct` field name mismatch (compounds F-1)

`backend/slack_bot/formatters.py:322`:

```python
total_return_pct = portfolio_data.get("total_return_pct", 0)
```

But every portfolio endpoint returns the key `total_pnl_pct`, not `total_return_pct`:
- Legacy `/api/portfolio/performance` (`portfolio.py:143`): `total_pnl_pct`
- Working `/api/paper-trading/portfolio` (`paper_trading.py:140`): `total_pnl_pct`

Even if F-1 is fixed and the digest hits the correct endpoint, the formatter reads the wrong key and returns 0. **Wrong P&L is a double bug** — wrong endpoint AND wrong key. Both must be fixed.

### F-3: `5x SNDK` recent-analyses bug

`backend/db/bigquery_client.py:258-268`:

```python
def get_recent_reports(self, limit: int = 10):
    query = f"""
    SELECT ticker, analysis_date, recommendation, confidence, target_price, summary
    FROM {self._reports_table()}
    ORDER BY analysis_date DESC
    LIMIT {limit}
    """
    ...
```

No `GROUP BY ticker`, no `QUALIFY ROW_NUMBER() OVER (PARTITION BY ticker ...)`, no Python-side dedup. If a single ticker (SNDK) was analyzed 5+ times since the last digest, all `LIMIT 5` rows are that ticker. The morning digest renders 5x SNDK rows in "Recent Analyses." Hypothesis confirmed.

### F-4: Morning digest 2:00 PM = wrong `.env` config (not code)

`backend/slack_bot/scheduler.py:144`:

```python
scheduler = AsyncIOScheduler(timezone=ZoneInfo("America/New_York"))
```

The APScheduler timezone is correctly `America/New_York` (DST-aware ET). The defaults in `backend/config/settings.py:199` are `morning_digest_hour=8`, `evening_digest_hour=17`. Operator-reported timings (2:00 PM and 11:00 PM) map to `.env` overrides `MORNING_DIGEST_HOUR=14`, `EVENING_DIGEST_HOUR=23`. **The "schedule bug" is operator config, not a code bug** — fix is a one-line `.env` edit (and ideally a settings validator + Slack-bot startup-log echo of effective cron times so future operators don't silently mis-config).

### F-5: Missing notification types (5 categories)

The bot infrastructure has `send_trading_escalation` at `scheduler.py:369-423` and `format_escalation_alert` at `formatters.py:624-686` with severity tiers (P0 iMessage + Slack, P1 Slack, P2 logger only). These are production-ready. Zero call sites wire the following:

**(a) Trade confirmation** — No `send_trade_confirmation` function exists. `paper_trader.execute_trade()` returns silently; `paper_trading.py:931` (cycle completion) does not post to Slack.

**(b) Kill-switch alert** — `pause_signals()` at `scheduler.py:353-366` calls `scheduler.shutdown()` + `logger.info` but does NOT call `send_trading_escalation()`. Operator never knows the kill switch fired.

**(c) Drawdown alarm** — `/api/paper-trading/drawdown` endpoint exists at `paper_trading.py:~363` but no threshold-check call site triggers an alert when realized drawdown breaches.

**(d) Cost-budget breach** — `backend/slack_bot/jobs/cost_budget_watcher.py` fires when BQ spend exceeds caps, BUT its `alert_fn` is only wired when `_production_fns.make_alert_fn_for_budget` is called during `register_phase9_jobs` at `scheduler.py:562-565`. If that wiring fails, the failure is logged fail-open at `scheduler.py:569-571` and cost alerts are silently dropped.

**(e) Cycle completion summary** — No Slack post fires when the autonomous daily cycle completes. Cross-references bucket 24.2 (pipeline routing) — if the full pipeline ran, that's notable. If only lite ran, less so.

**(f) Error escalation routing** — `logger.exception` sites in `scheduler.py` are not promoted to `send_trading_escalation("P1", ...)`. Backend errors are visible only in `handoff/logs/`.

**(g) Weekly autoresearch summary** — `backend/meta_evolution/cron.py` runs Sunday-only (bucket 24.3) but no Slack summary fires when it completes. Operator has no signal whether the weekly run succeeded or what it changed.

### F-6: `/portfolio` slash command reproduces wrong-endpoint bug

### Source 3: OneUptime — Alert Deduplication (Jan 2026)

Fix: `QUALIFY ROW_NUMBER() OVER (PARTITION BY ticker ORDER BY analysis_date DESC) = 1` is a SQL-level dedup that mirrors the OpsGenie pattern.

### Source 4: Stockalarm — Day Trading Alerts (2026)

5 essential trading alert types; urgency tiers (phone > push > text > email); start with 5-10 high-quality alerts not 50; daily hygiene — deactivate closed-position alerts immediately. **Application:** when wiring the missing notification categories (F-5), start with 5 highest-value: trade confirmations, kill-switch, drawdown, cost-budget, cycle-complete. Add weekly autoresearch summary only after the first 5 are validated.

### Source 5: Anthropic — Harness Design for Long-Running Apps

File-based handoff as implicit audit trail; contract-based verification; evaluator as behavioral health monitor. **Application:** Slack notifications are the operator-facing audit trail for the autonomous trading system.
