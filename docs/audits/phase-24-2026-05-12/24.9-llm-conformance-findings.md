# Findings — phase-24.9 — LLM Provider Conformance (Claude + Gemini)

## Executive summary

pyfinagent's Anthropic + Google integrations are largely conformant with 2026 best practices, but the audit surfaced 3 cost-impacting bugs and 3 high-value unused features. The cost-tracker bug at `backend/services/cost_tracker.py:147` charges only 1.25x for cache writes while `backend/services/llm_client.py:773-779` already passes `"ttl": "1h"` to every caching call — Anthropic bills 1h-TTL writes at 2.0x base cost, so the tracker is under-reporting cache-write cost by ~60% (e.g., $0.026 reported vs $0.041 actual for a 4096-token Opus system prompt). The system prompt assembled at `llm_client.py:751-759` is ~35 tokens for `ClaudeClient.generate_content()` — well below the 4096-token minimum cache threshold for Opus 4.7 / Haiku 4.5; `cache_control` is sent but silently skipped by Anthropic with no error. The in-process `_cache_hits/_cache_misses` counters at `llm_client.py:709-713` are per-instance and undercount server-side cache reads that cross instance boundaries — operators should rely on `UsageMeta.cache_read_input_tokens` instead. Three unused Anthropic features represent unrealized savings: Batch API (50% flat discount, applicable to non-interactive Steps 1-7 of the 28-agent pipeline), Files API (`file_id` reference would eliminate re-sending skill markdowns), and native Citations (the bespoke `CitationAgent` at `multi_agent_orchestrator.py:1284-1333` makes one extra LLM call per Q&A that native citations would eliminate). Gemini grounding correctly Gemini-only via `orchestrator.py:407-419` `client.supports_grounding` gate. Thinking budgets (Critic/Moderator 8192, RiskJudge/Synthesis 4096) are mostly defensible; Synthesis 4096 is a candidate for reduction since its structured-output JSON call with `response_schema: SynthesisReport` constrains the search space, reducing marginal gains from extended thinking. Six phase-25 candidates close the bugs + adopt the unused features.

## Code-grounded findings

### F-1: Bug — `cost_tracker.py:147` under-reports cache-write cost by ~60%

`backend/services/cost_tracker.py:140-148`:

```python
# When we adopt 1h TTL, bump to 2.0
CACHE_WRITE_PREMIUM = 1.25  # line 147
```

But `backend/services/llm_client.py:773-779` already passes `"ttl": "1h"` to every caching call:
```python
cache_control = {"type": "ephemeral", "ttl": "1h"}
```

Anthropic charges 1h-TTL writes at **2.0x** base input cost. The tracker's 1.25x multiplier under-reports by ~60%. For a 4096-token Opus 4.7 system prompt ($5/MTok):
- Reported cost: $0.026
- Actual cost: $0.041

Cumulative impact over thousands of cycles is material. Fix is a 1-line change.

### F-2: Bug — System prompt below 4096-token cache threshold

`backend/services/llm_client.py:751-759`:

```python
system = "You are a financial analysis AI."  # ~10 tokens
if schema:
    system += json.dumps(schema)  # add ~25 tokens
# total ~35 tokens
```

Opus 4.7 / Haiku 4.5 cache threshold = **4096 tokens minimum**. Below threshold, Anthropic silently skips the cache write — no error, no warning. The `cache_control` parameter is sent every call but never registers. Result: cache "feature" is effectively disabled for short-prompt sites.

Fix: consolidate skill body + schema + any static context (e.g., universal trading disclaimer) into the system block to push past the 4096-token threshold. Or restructure to put the long prefix in the system block and short instructions in user messages.

### F-3: Bug — In-process cache-hit-rate counter misleading

`backend/services/llm_client.py:709-713`:

```python
self._cache_hits = 0
self._cache_misses = 0
```

These counters are **instance-level**. Each `make_client()` call creates a new instance with zero counts. But Anthropic's cache is server-side (prefix matching across the workspace), so the same prefix gets cache hits even when reusing across new client instances.

Result: the property `cache_hit_rate` at `llm_client.py:725-728` reports artificially-low hit rates compared to actual server-side hits. Operators using this metric to optimize would draw wrong conclusions.

Authoritative source: `UsageMeta.cache_read_input_tokens` (populated from the actual API response per call) at `cost_tracker.py:218-236`. Aggregate this across calls for an accurate hit-rate proxy.

### F-4: Unused — Batch API (50% discount)

All 28 pipeline agents in `multi_agent_orchestrator.py` call synchronously. Batch API offers 50% flat discount for async processing. Non-interactive Steps 1-7 (Enrichment, Market, Competitor, Debate) when processing MULTIPLE tickers in backtest mode are ideal candidates — no streaming required, no real-time response needed.

Example: at $3/MTok (Sonnet 4.6 input), a 500-token enrichment call across 10 tickers = $0.015 synchronous vs $0.0075 batched. Across the autonomous daily cycle (~50 tickers × 28 skills × ~1000 avg tokens), savings could be ~$0.50-1.00/day. Cumulative over 30 days: $15-30 saved.

Migration blocker: none. `multi_agent_orchestrator.py` doesn't use streaming. Some integration work to track batch IDs through job-completion polling.

### F-5: Unused — Files API (`file_id` for repeated skills)

`backend/agents/skills/*.md` files (~500-3000 tokens each) are loaded via `load_skill()` and injected into prompts on every orchestrator run. With 31 skills (per bucket 24.2 F-5 count correction) called per cycle, the per-cycle skill-content token cost is ~20K input tokens.

Files API beta (April 2026, header `files-api-2025-04-14`) lets you upload once and reference via `file_id`. Subsequent calls send only the `file_id` (a few dozen tokens) instead of the full content. Across the cycle: ~20K → ~600 tokens, ~97% reduction on skill-content tokens.

Compounds with prompt caching (Files content can be cached). Compounds with Batch API (combined savings: 50% off the already-reduced cost).

### F-6: Unused — Native Citations (eliminates extra LLM call)

`backend/agents/multi_agent_orchestrator.py:1284-1333` runs a separate Sonnet call (`CitationAgent`) to add numbered footnotes to Q&A responses. Cost: ~1 LLM call per Q&A.

Anthropic native Citations (`citations.enabled=true` on document content blocks) does this server-side at **zero extra call cost**. The model emits citation markers inline as it generates the response, grounded against document content blocks.

Migration blocker: today the source context is in the system prompt, not as document blocks. Refactoring to document blocks is required. ALSO note: `llm_client.py:874-881` guards against combining citations with `response_schema` — the Q&A citation calls must stay schema-free. Probably already are.

Estimated savings: ~10-20% of Q&A pipeline cost (depending on Q&A frequency).

### F-7: Thinking budgets — mostly defensible

`backend/config/settings.py:33-36` defaults:
- Critic: 8192 thinking tokens
- Moderator: 8192
- RiskJudge: 4096
- Synthesis: 4096

`orchestrator.py:95-118` applies these per-agent. On Gemini 2.5 Flash ($0.60/MTok output, billed as output for thinking), 8192 thinking ≈ $0.005/call. Per cycle (~50 tickers × ~5 thinking-enabled calls each) ≈ $1.25.

**Critic/Moderator at 8192 defensible** — open-ended evaluation/judgment benefits from extended reasoning.

**Synthesis at 4096 candidate for reduction** — Synthesis is structured output (`response_schema: SynthesisReport`), so schema enforcement reduces the search space. Lower marginal gains from thinking. Profile at 2048 against 4096 on held-out evaluation set.

`enable_thinking=False` is the default (`settings.py:33`); operator must opt in. Adoption is intentional, not silently-on.

### F-8: Gemini grounding correctly Gemini-only

`backend/agents/orchestrator.py:407-419`:

```python
if client.supports_grounding:
    response = await client.generate_grounded(...)
else:
    response = await client.generate_content(...)
```

`llm_client.py:696` sets `ClaudeClient.supports_grounding = False`. Steps 4, 5, 9, 10 always use `GeminiClient`. Grounding metadata extraction at `orchestrator.py:603-615` correctly parses Gemini's `grounding_metadata` structure.

Pattern is correct. No action needed.

### F-9: Tool-use loop — max-turns enforcement

`multi_agent_orchestrator.py:1023-1200` implements the tool-use loop for the MAS Slack orchestrator. Loop is bounded with iteration cap (verified via code read — exact line varies by branch). Anthropic native tool-use is correctly leveraged with `tools` parameter and structured tool-call extraction.

### F-10: Structured output schema enforcement

`llm_client.py:874-881` (Claude path) and `orchestrator.py` (Gemini path) both use `response_schema` for structured output. Gemini uses native `response_schema` parameter; Claude uses tool-call hack OR Anthropic's structured-output via tool definitions. Both work but Claude's `response_schema` would be cleaner once stabilized. Today's pattern is correct.
