### Even with everything healthy, "Approve" approves nothing (pyfinagent side)

- The one real HITL flow — monthly champion/challenger — takes approval via API (`backend/api/monthly_approval_api.py`), not channel text.

### Fail-open or fail-closed?

- **The approval mechanism itself is fail-closed.** Anything genuinely waiting for approval stays blocked: the champion/challenger HITL window auto-expires pending approvals after 48h with no promotion (`backend/autoresearch/monthly_champion_challenger.py:10-13`, `:108-117` — `prior["status"] = "expired"`). No approval was granted by the broken path; nothing interpreted failure as consent; no unauthorized action occurred.
- **The system's operating posture under approval-path failure is fail-open autonomy.** Nothing in the trading loop requires operator approval (`autonomous_loop.py` has no approval gate; 33 trades executed 06-01..06-10). The broken channel did not stop trading — it removed the operator's only remote control surface during the only week he was away. The same dead path that ate "Approve" would have eaten "pause" or "stop". The operator's successful interventions that day (manual pause/resume pairs 06-01 15:47, per the 55.1 kill-switch audit trail) went through the web UI, not Slack.

Severity: **P0** in the away-week context, with the fail-closed caveat above. Fix shape (phase-56.2): repair gateway anthropic auth AND implement a deterministic approval consumer (text grammar or wired button) that does not require an LLM turn to register an approval, AND alert when channel-message ingestion is silent for >N days (a 6-week ingestion outage was invisible).

## 2. Incident: nightly backend-watchdog "unreachable/ReadTimeout" (05-27, 05-28, 06-04)

### What emits it

Not the bash watchdog. The string matches `backend/slack_bot/scheduler.py:493` — `detail = f"unreachable: {type(exc).__name__}"` → "unreachable: ReadTimeout", posted as ":rotating_light: Watchdog Alert -- Backend unreachable" at `:515` by `_watchdog_health_check`, which probes `http://127.0.0.1:8000/api/health` with `httpx` timeout=10.0s (`:485`) every 15 min (`settings.py:504`), posting only on state transitions (`:497-528`). A second, separate watchdog exists: the launchd bash script `scripts/launchd/backend_watchdog.sh` (curl `-m 5`, every 60s, 3 consecutive fails → SIGUSR1 stack dump + `launchctl kickstart -k`).

### Exact incident windows (primary, Slack MCP)

| date | Alert (CEST) | Recovery (CEST) | duration | cycle that day (CEST) |
|---|---|---|---|---|
| 05-27 | 20:14:11 | 20:29:01 | 1 probe (15 min) | 18:59 start; SIGTERM-killed 19:48:49; re-run 19:49→~20:30 (the 0.0 burst 20:02-20:20) |
| 05-28 | 20:05:13 | 20:20:03 | 1 probe | 20:00:45 start; (separate SIGTERM restart later at 20:39:31, AFTER recovery) |
| 06-04 | 20:35:19 | 20:50:08 | 1 probe | 20:02:06 start → heartbeat 21:02; no restart |

Each "incident" was exactly ONE failed 10s-budget probe on the 15-minute cadence, recovered on the next probe. The "~20:05-20:50" range in the operator's recollection is the union of three different single-probe windows across three days — every one inside the daily trading cycle.

### Trigger: the daily trading cycle starves the event loop

`httpx.ReadTimeout` means the TCP connection was ACCEPTED but no response arrived within 10s — process alive, asyncio loop not serving. This is a known, previously diagnosed failure mode — the code says so itself: `backend/services/autonomous_loop.py:716-723` (phase-23.1.23 comment: mark_to_market's blocking ops "previously blocked /api/health past the watchdog threshold and got the backend kickstart-killed daily"). The mitigation wrapped *some* blocking calls in `asyncio.to_thread`, but blocking work remains on the loop: the lite analyzer calls `yf.Ticker()` / `.info` / `.history()` synchronously inside `async def _run_claude_analysis` (`autonomous_loop.py:1492-1494`; same in `_run_gemini_analysis` at `:1744-1746`), and that week every cycle additionally ran full-orchestrator attempts that hang up to 90s per agent before timing out (section 4).

### Was the backend actually down?

**No.** Restart census from backend.log (every `faulthandler registered` line = process start, date-anchored via embedded APScheduler timestamps):

```
2026-05-27 18:59:25, 19:48:52, 21:03:49
2026-05-28 20:39:31, 23:51:56
2026-05-29 00:23:20, 22:47:44, 22:48:31, 23:11:57
2026-06-01 01:42:42
(next) 2026-06-11 05:03:45
```

- **06-04:** zero restarts — the backend ran uninterrupted across the whole away week (06-01 01:42 → 06-11 05:03). The 06-04 cycle completed normally (trades 21:00-21:01 CEST, heartbeat 21:02, snapshot written). Pure starvation; one missed probe.
- **05-27 / 05-28 (pre-away, operator at home):** graceful mid-cycle SIGTERM restarts — 05-27 19:48:49 "Shutting down" while Deep Dive/Synthesis were in flight, followed at 19:49:01 by `POST /api/paper-trading/run-now` from localhost (a human at the dashboard); 05-28 20:39:27 "Shutting down" 39 min into the cycle (stale-lock age 2370s at restart), *after* that evening's watchdog alert had already recovered. Clean SIGTERMs, not crashes, not watchdog kills.
- **The bash watchdog never fired in this period:** zero SIGUSR1 stack dumps in backend.log after 2026-05-04, and no kickstart-pattern (SIGKILL, no graceful shutdown lines) restarts. The starvation was bursty (>10s blockages, never 3 consecutive 60s-spaced curl failures).

Verdict: trigger identified (scheduled-cycle event-loop starvation; residual sync I/O plus that week's 90s hang-then-timeout loops), compounded on 05-27/05-28 by operator/dev mid-cycle restarts. The alert cannot distinguish "down" from "busy", and the operator reasonably read three one-probe blips as nightly instability. P2 for alert semantics; the starvation itself is P1 because it keeps the bash watchdog one threshold away from SIGKILLing a live trading cycle.

The Gemini sub-agents kept working (88 gemini-2.0-flash rows in `llm_call_log` on 05-27) — full enrichment context, dead Claude synthesis tier.

## 6. Reasoning-quality spot-checks (3+ stored analyses, including whipsaw trades)

### Spot-check A — MU 06-08 BUY → 06-09 forced exit (the −6.3% one-day round trip; whipsaw trade)

- **Entry reasoning** (`analysis_results` 06-08 18:10 UTC, BUY 7.0, conf 68): "All momentum and market-cap gates clear with no clear negative signal, though the +124% 60-day run and 45x P/E warrant a tight trailing stop against mean-reversion risk." Risk judge: volatility EXTREME, APPROVE_REDUCED sizing. Internally consistent with its (price-only) inputs; every number traceable to the logged m20=+20.1%, m60=+124.2%, P/E 45.1. Honest about reversal risk.
- **The exit had no reasoning at all.** There is **no MU analysis row on 06-09** (cycle log: "Analyzing 5 new + 0 re-evals"). Mechanism: `execute_buy` stamps `last_analysis_date=now` (`paper_trader.py:303,327`); re-eval fires only at `days_since >= 3` (`autonomous_loop.py:727-740`; `settings.py:308`); the swap path then treats any holding absent from the same-cycle `holding_lookup` as **conviction 0.0** — the code says it plainly: "No fresh analysis => unknown conviction. **Treat as worst**" (`portfolio_manager.py:443-449`); swap delta vs a 0.0 sentinel = `cand_score/0.01*100` (`:497-498`) — astronomically past the 25% threshold for any candidate. MU, bought 24h earlier at the cycle's top score, was sold as "swap_for_higher_conviction" for a candidate with the *same* score (DELL 7.0). The label is false: it was swap-vs-unknown. The identical mechanism produced the STX/DELL exits on 06-08 (bought 06-05, 2d23h < 3 days → sentinel), the SNDK exit on 06-09, and the DELL exit on 06-10 — the away week's churn engine. **Verdict: no decision-quality reasoning existed for the money-losing leg.** (Bonus defect: `_compute_swap_candidates`'s comment believes "final_score lives in [0,1]" (`portfolio_manager.py:495-496`) while the lite path emits 1-10 — the 25% relative threshold is mis-scaled even for *fresh* holdings: any 7-vs-5 integer difference = +40% "uplift" and fires.)

### Spot-check B — SNDK 06-05 HOLD vs 06-08 BUY (the flip)

- **Factual traceability:** good *within its tiny input set*. Every number in the stored reasons matches the logged `market_data` inputs; no invented news, filings, or events anywhere in the away-week rationales (the no-hallucination guardrails appear effective on that axis).
- **But the inputs are suspect and the model said so, to no effect.** 06-03 SNDK risk judge: "implausible price/market-cap data (**likely a bad bar**) make this a reversion/data-quality trap" — SNDK priced at $1,818/$1,553/$1,632 with a $230-270B cap across the week. There is no path from "the model flagged its own inputs as corrupt" to any action; the flag lives and dies in the `summary` column.
- **Calibration:** poor. Confidence collapses to a 58-72 band (62 on 7+ rows — anchoring near the prompt's example value); confidence *fell* 72→68 across the HOLD→BUY flip; integer scores 3-8 carry ~no resolution. The 05-28 all-0.0 day is the canonical miscalibration: a scoring outage rendered as confident-looking "0.0/10 HOLD" rows, schema-indistinguishable from honest scores.

### Spot-check C — 000660.KS 06-04 re-entry BUY 7.0 (after 06-02 swap-out; stopped out 06-05 at −9.9%)

- Stored trader reason (06-04 18:04 UTC): "Momentum rule cleared decisively (+58.8% 20d, +148.7% 60d, mega-cap), but the move is extended and P/E=0.0 is missing not reasonable, so confidence is tempered." Risk judge: "VALUATION (UNRELIABLE): P/E of 0.0 and a **nonsensical $1.63-quadrillion market cap are corrupted/missing data**... approve only a heavily reduced 2% starter."
- **The prompt itself was corrupted:** the lite analyzer formats KR inputs as `Price: $2298000.00 | Market Cap: $1630000.0B` (`autonomous_loop.py:1553-1557` renders raw yfinance KRW values with `$` literals) — the same KRW-as-USD defect class 55.1 confirmed in the trade ledger (B1/B2), here injected into the LLM's decision input. The model partially compensated (it knows SK hynix is a mega-cap) and still recommended BUY into a +149% 60d parabola one day before a −9.9% stop-out, on a KR book whose momentum history the system had only just begun observing through its own FX-mangled lens (KR enabled 06-01).
- **Verdict:** the reasoning is honest about its garbage inputs, but the system design forces a verdict anyway; "tempered confidence" (62) is not a risk control when the sizing floor still buys.

### Spot-check D (structural, all three trades above) — the Risk Judge is decorative at the admission gate

`paper_trades` rows: **06-02 HPE BUY with `risk_judge_decision=REJECT`; 06-03 DELL BUY with REJECT** (judge: "entering a near-tripling parabolic move at a stretched multiple is buying the top — **reject the fresh BUY**"); **06-09 066570.KS BUY with REJECT**, where the judge's stored rationale correctly diagnosed the engine's own FX bug: "a $44,540.6B (~$44.5T) market cap is **physically impossible for LG Electronics (a KRW/USD unit error on the newly-onboarded KR market)**" — and the BUY executed anyway; stopped out 06-10 at −9.7% (₩248,000 → ₩224,000). Root cause at file:line: the buy-candidate loop admits on the *trader's* recommendation only (`portfolio_manager.py:150-157`, `rec not in _BUY_RECS → continue`); the judge's `decision` is logged (`:193-198`) and persisted, but **no code path anywhere checks `decision == "REJECT"`** — only its `recommended_position_pct` flows into sizing (`:288`). The single most damning quality finding of the week: the only agent that was right — about the top-chasing and about the data corruption — is structurally ignored at the only gate that mattered. ("Did the cited skills' outputs actually inform the decisions": for the risk-judge skill — *sizing yes, verdict no*.)
