# Fix: Phase 4.5 widgets bypass apiFetch — route to wrong host

**Drafted:** 2026-04-16
**Parent:** `handoff/current/phase-4.5-contract.md`
**Cycle type:** post-release bug fix (not a masterplan step; tracked in harness_log for audit)
**Research tier:** trivial (root-cause is one line of evidence)

## Hypothesis
Several Phase 4.5 widgets use raw `fetch("/api/paper-trading/...")` calls that hit the frontend host (:3000) instead of the FastAPI backend (:8000). On :3000 those paths are not registered Next.js routes, so the middleware sees non-allowlist paths and either redirects to /login (302) or Next serves 404. Replacing the raw fetches with the existing `apiFetch()` helper (which uses `API_BASE = process.env.NEXT_PUBLIC_API_URL || "http://localhost:8000"` and handles auth + timeouts + error shapes) routes the requests correctly.

2. Kill Switch panel renders the status banner + buttons (not the empty skeleton).
3. Agent rationale drawer loads on trade click.
4. Live intraday prices populate the Positions tab Current column.
5. MFE/MAE scatter loads on the Exit-quality tab.

7. Type check clean; shell-class assertion holds; 18/18 pytest.

- No per-component auth-header plumbing (centralized in `apiFetch`).
