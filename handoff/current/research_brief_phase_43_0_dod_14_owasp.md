## 4. Recency scan (2024-2026)

Searched scoped queries `"OWASP LLM" "2026" version release update errata since 2025`, `OWASP LLM04 data poisoning RAG retrieval no fine-tuning 2026`, `LLM hallucinated ticker symbol detection trading agent 2026`. Findings:

- **No new OWASP LLM Top-10 list since March 2025.** Authoritative sources (Repello AI, alexewerlof, Straiker, LinkedIn post by Steve Wilson) all confirm the 2025 list (LLM01–LLM10) remains current. OWASP published THREE separate 2026 landscape documents in Q2 2026 — (a) updated **Agentic** Top-10 (ASI prefix, December 2025 release), (b) Q2 2026 LLM/GenAI Applications Landscape (taxonomy of solutions, NOT a new Top-10 list), (c) first-ever Red Teaming Landscape (April 10, 2026). None of these supersede the 2025 LLM Top-10 list.
- **No errata or revised definitions for LLM04/05/09.** Canonical text at `genai.owasp.org/llmrisk/llmXX2025-<slug>/` matches what was published in March 2025.
- **Trading-domain LLM09 research (2025-2026):** TrustTrade paper (arXiv 2603.22567, late 2025) and CLAP paper (Cross-Layer Attention Probing, September 2025) advance hallucination detection via consensus / activation monitoring. These complement rather than supersede the OWASP LLM09 framing. The actionable insight for SKILL.md is the validation-layer architecture (deterministic price/volatility grounding for trading signals) — not a new heuristic but a confirmation that the proposed `llm-output-to-execution-without-validation` BLOCK heuristic is the right framing.
- **No supersession of the existing `rag-memory-poisoning` LLM08 heuristic in SKILL.md.** arXiv 2505.05196 (May 2025) confirms RAG embedding-corpus poisoning remains active research; pyfinagent uses BM25 (lexical) not vector embeddings, so the Vec2Text inversion subset of LLM08 is N/A, but the *poisoning* subset (adversary-controlled documents entering the corpus) IS the active surface — already covered.

### Edit C — LLM05 explicit tag + sub-bullets (Dimension 1 security table line 81)

**Current text (line 81):**
```
| insecure-output-handling | `llm_call(...)` result flowing directly into `query(...)`, `exec(...)`, or file path | BLOCK |
```

**Replacement text:**
```
| insecure-output-handling [LLM05:2025] | `llm_call(...)` / agent-response string flowing directly into any of: (a) **command-injection** sinks: `exec()`, `eval()`, `subprocess.run(shell=True)`, `os.system()`; (b) **SQL injection** sinks: `cursor.execute(...)` / `db.query(...)` with f-string or string-concat instead of parameterized query; (c) **path traversal** sinks: `open(...)`, `Path(...)`, file-write or file-read where the path string contains an LLM-output substring; (d) **SSRF** sinks: `requests.get(...)`, `httpx.get(...)`, `urllib.request.urlopen(...)` with an LLM-generated URL; (e) **XSS / HTML injection** sinks: response HTML / template render / FastAPI JSONResponse containing un-escaped LLM output. Grep: `(exec\\|eval\\|os\\.system\\|subprocess.*shell=True)\\b.*\\b(llm_\\|agent_\\|response\\|completion)`, `cursor\\.execute.*f["']`, `open\\(.*llm_`, `requests\\.(get\\|post).*llm_` | BLOCK |
```

**Rationale:** Five sub-sinks per OWASP LLM05:2025 canonical text + StackHawk + FireTail + Indusface — all five sources cited the same enumeration (with slight phrasing differences). Severity BLOCK is unchanged and correct per StackHawk severity rubric ("Critical" for exec/eval; "High" for SQL/HTML/file ops; we collapse to a single BLOCK because pyfinagent's risk profile justifies the conservative call). Confidence: high.

### Edit D — LLM09 new BLOCK heuristic (Dimension 2 trading-domain table)

**Add a new row to the Dimension 2 — Trading-domain correctness table (after `paper-trader-broad-except` at line 118, before the negation list):**

```markdown
| llm-output-to-execution-without-validation [LLM09:2025] | NEW code path where LLM-generated content (agent response, signal recommendation, ticker selection, target-price suggestion) reaches **trade execution** (`paper_trader.execute_buy()` at `paper_trader.py:85`, `paper_trader.execute_sell()` at `paper_trader.py:299`) **OR** **signal emission** (`agents/mcp_servers/signals_server.py`, `services/pead_signal.py`, `services/insider_signal_screen.py`, `services/defense_signal.py`) **without** an intermediate validation step. Required validation = (a) ticker exists in `pyfinagent_data.historical_prices` (verifies non-hallucinated symbol); (b) any price/volume/fundamental cited resolves to a BQ `historical_*` row within ±1 trading day; (c) any cited 10-K / filing reference resolves to a real `financial_reports.*` row; (d) confidence framing on speculative LLM output is bounded (no "guaranteed" / "certain" language passed to execution). Grep: agent/LLM-response string flowing into `execute_buy(` or `execute_sell(` call within the same function with NO intervening call to `perf_metrics`, `bigquery.Client`, `historical_prices`, or a known validator function. | BLOCK |
```

**Add to the Dimension 2 negation list (after the existing `crypto-asset-class` bullet near line 124):**

```markdown
- `llm-output-to-execution-without-validation`: existing call sites that go through the deterministic signal pipeline (`pead_signal.py`, `insider_signal_screen.py`, `defense_signal.py`) ARE the validators — they emit candidates that are checked against `historical_prices` BQ tables and `perf_metrics.py` before reaching `paper_trader.execute_buy()`. Do NOT flag the current pipeline. ONLY flag NEW code paths that bypass these validators (e.g., a hypothetical "ask Claude what to buy and execute directly" path).
```

**Rationale:** LLM09 Misinformation is the highest-leverage 2025-list risk for pyfinagent because LLM output flows into real trade execution. OWASP LLM09 canonical text recommends "automatically validate key outputs, especially output from high-stakes environments" — trade execution IS high-stakes. The four validation sub-checks (ticker existence; price/volume anchor; citation resolution; confidence framing) directly map to the OWASP hallucination taxonomy (hallucinated entities, fabricated facts, fabricated citations, misrepresentation of expertise/confidence). The TrustTrade paper (arXiv 2603.22567) independently validates the "deterministic temporal-signal grounding" pattern as the correct architecture between LLM output and execution. Severity BLOCK is correct because an unvalidated LLM-to-execution path is a financial-loss vector, not a degraded-quality vector. Confidence: high.
