## 1. Incident triage (criterion 1)

### F-A1 — Slack "Approve" → `Missing API key for provider "anthropic"` (severity HIGH)

**Root cause at file:line.** The literal error string exists NOWHERE in repo or venv code (exhaustive grep; litellm's nearest string is "Missing API key for **Volcengine**"). It is emitted by the **`claude` CLI binary** invoked through `backend/agents/claude_code_client.py::claude_code_invoke()`. That wrapper **deliberately scrubs `ANTHROPIC_API_KEY`/`ANTHROPIC_AUTH_TOKEN` from the subprocess env** (`claude_code_client.py:163-170`, phase-38.13.1) so the CLI authenticates via the `~/.claude/` OAuth session (Max flat-fee rail) instead of metering api.anthropic.com. With the OAuth session invalid during the unattended window, the CLI had neither an env key (scrubbed by design) nor a token → the provider-named missing-key error, wrapped as `ClaudeCodeError` (`claude_code_client.py:188-197`) and relayed verbatim to Slack by the assistant handler's exception path.

**Message route:** operator's "Approve" text → `commands.py:185 handle_any_message` (every non-bot message in the approvals channel) → ticket ingest + assistant/orchestrator answer path → `make_client()` → `ClaudeCodeClient` (because `paper_use_claude_code_route=True`, confirmed live). "Approve" is not in `direct_responder.py:33-46`'s no-LLM fast-path triggers.

**Cross-evidence:** (i) 4 **direct-Anthropic** Haiku probes at 06-01 15:31-15:46 UTC succeeded (`llm_call_log`, ok=true) — the env API key WORKED at that moment; (ii) the kill-switch audit log shows two manual pause→resume pairs at 06-01 15:47 (the operator's interaction window); (iii) every away-week BUY carries the SignalStack fallback string (§2) and `llm_call_log` has zero rows after 06-01 19:11 — the CLI rail stayed down for the whole week while the direct-API rail was alive. (iv) Live now (post-return): `claude auth status` → `loggedIn: true, authMethod: claude.ai, apiProvider: firstParty` — the rail recovered with the operator's presence, consistent with an OAuth-session/keychain issue in the unattended/headless context rather than a config regression.

**Triage:** the away-week failure is bounded to **OAuth-session validity of the `claude` CLI rail in unattended operation**; the `.env` key was present and valid the whole time. **Fix direction for 56.2 must NOT be "un-scrub the key"** — that silently re-bills the metered account (the exact thing phase-38.13.1's guard at `llm_client.py:1967` exists to prevent). The right 56.2 shape: detect CLI-auth failure at cycle start, alert loudly to Slack, and (operator-gated) define the degraded-mode policy. The same missing-key family applies to the failing `claude-review` CI check (no `ANTHROPIC_API_KEY` secret in GitHub Actions — separate surface, same family, operator-gated to fix).
