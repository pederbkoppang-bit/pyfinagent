## 1. Incident: Slack "Approve" → "Missing API key for provider anthropic" (06-01, x2)

### Root cause chain (file:line)

The reply did not come from pyfinagent's Slack bot. It came from the **OpenClaw gateway**, which is bound to the same channel: `~/.openclaw/openclaw.json` has `bindings[0].match.peer.id = C0ANTGNNK8D` routed to `agentId: pyfinagent`.

1. Operator's "Approve" message → OpenClaw routes it to its embedded `pyfinagent` agent, which needs an Anthropic model turn just to process the inbound message (configured model `anthropic/claude-sonnet-4-20250514` per the fallback-decision log lines).
2. Auth resolution fails and throws at **`/opt/homebrew/lib/node_modules/openclaw/dist/model-auth-CElc27BR.js:310`**:
   `No API key found for provider "anthropic". Auth store: /Users/ford/.openclaw/agents/pyfinagent/agent/auth-profiles.json ...`
3. The user-facing string is built at **`/opt/homebrew/lib/node_modules/openclaw/dist/agent-runner.runtime-CH0aH7T6.js:544`** (`buildMissingApiKeyFailureText`, provider in `SAFE_MISSING_API_KEY_PROVIDERS` at :533): `"Missing API key for provider \"anthropic\". Configure the gateway auth for that provider, then try again."` — byte-for-byte the message the operator saw.

### Log evidence (the two attempts)

`~/.openclaw/logs/gateway.err.log:5792-5799` — two failures on lane `session:agent:pyfinagent:slack:channel:c0antgnnk8d`:
```
2026-06-01T16:51:56.737+02:00 lane task error ... No API key found for provider "anthropic" ...
2026-06-01T16:51:58.283+02:00 lane task error ... (same)
```
`gateway.log:25511-25512` — the two error replies delivered to `channel:C0ANTGNNK8D` at 16:51:57 and 16:51:58 CEST. Exactly two events = the operator's two "Approve" messages.

### How long has it been broken

First occurrence in `gateway.err.log` is **2026-04-09 19:09** (line 121); the most recent is **2026-06-11 05:03** (line 6073+). It is a standing misconfiguration, not an away-week transient. Auth-store state (booleans only): `~/.openclaw/agents/pyfinagent/agent/auth-profiles.json` exists (mtime 2026-04-09 19:07), contains profile `anthropic:default` of type `token` with a non-empty token — yet OpenClaw's resolver (custom-key → auth-profile → env → plugin chain in `model-auth-CElc27BR.js:180-314`) rejects it and falls through to the throw. The `main` agent lane fails identically, so this is gateway-wide for the anthropic provider. **Honest bound:** the precise reason the non-empty `token`-type profile fails to resolve is inside OpenClaw's bundled `resolveApiKeyForProfile`; what is certain from logs is that no usable anthropic credential resolves for any agentDir, in every attempt logged April 9 → June 11. (Likely class: the stored token is a credential type the gateway cannot use for direct API calls; fixing is `openclaw agents add` / re-auth per the error's own remediation text. The error text the operator saw also tells the operator exactly this — but only if read as an ops alert rather than noise.)

### Was there even a consumer for "Approve"? (pyfinagent side)

No. Even with a healthy gateway or healthy pyfinagent bot, typing "Approve" approves nothing:

- `backend/slack_bot/commands.py:184-273` (`handle_any_message`) is the only channel-text handler: it ingests the message as a ticket and routes it to a generic LLM responder (`backend/services/ticket_queue_processor.py:137-264` — direct `anthropic.Anthropic` call). There is **no text grammar for approvals anywhere** (grep for approve/Approve across `backend/` finds no matcher).
- The Block-Kit "Approve" button (`backend/slack_bot/governance.py:165-168`, `action_id="approval_approve"`) has **no registered `@app.action` handler** (the only `app.action` registrations are in `app_home.py:397-452`), and its sender `HumanInTheLoopManager.send_approval_gate` is **never invoked** anywhere in the codebase. Dead UI.
- The only working Slack approval is the emoji-reaction git-push approval (`commands.py:275-302`).
- The one real HITL flow — monthly champion/challenger — expects its approval via API (`backend/api/monthly_approval_api.py`), not via channel text.

### Fail-open or fail-closed?

Two-part answer, both directions documented:

- **The approval mechanism itself is fail-closed.** Anything that genuinely waits for approval stays blocked: the monthly champion/challenger HITL window auto-expires a pending approval after 48h with no promotion (`backend/autoresearch/monthly_champion_challenger.py:10-13` and `:108-117`, `prior["status"] = "expired"`). No approval was ever granted by the broken path, and nothing interpreted the failure as consent. No unauthorized action occurred.
- **The system's operating posture under approval-path failure is fail-open autonomy.** Nothing in the autonomous trading loop requires operator approval (`backend/services/autonomous_loop.py` has no approval gate; trades executed all week: 33 paper_trades rows 06-01..06-10). The broken gateway therefore did not stop trading — it removed the operator's only remote control surface during the only week he was away. The same dead channel that ate "Approve" would have eaten "pause" or "stop". From the operator-control perspective the away-week system was un-commandable, and proceeded.

**Honest bound on what was being approved:** the gateway logs do not record inbound message text, so the specific request the operator answered cannot be recovered from primary data here. The 55.1 kill-switch audit records two manual pause/resume pairs at 06-01 15:47 (UI-triggered), i.e. the operator did succeed via the web UI roughly two hours before the failed Slack attempts at 16:51 CEST — consistent with Slack being tried as the remote channel and failing.

Severity: **P0** for the away-week context (single remote control channel dead), with the fail-closed caveat above. Fix shape (for phase-56.2): repair gateway anthropic auth AND implement an actual deterministic approval consumer (text grammar or button handler) that does not require an LLM turn to register an approval.

## 2. Incident: nightly backend-watchdog "unreachable/ReadTimeout" ~20:05-20:50 CEST (05-27, 05-28, 06-04)

### What emits it

Not the bash watchdog. The string matches `backend/slack_bot/scheduler.py:493` — `detail = f"unreachable: {type(exc).__name__}"` → "unreachable: ReadTimeout", posted as ":rotating_light: Watchdog Alert -- Backend unreachable" at `:515` by `_watchdog_health_check`, which probes `http://127.0.0.1:8000/api/health` with `httpx` timeout=10.0s (`:485`) every 15 min (`settings.py:504`, `watchdog_interval_minutes=15`), posting only on state transitions (`:497-528`). There is a second, separate watchdog: the launchd bash script `scripts/launchd/backend_watchdog.sh` (curl `-m 5`, every 60s, 3 consecutive fails → SIGUSR1 stack dump + `launchctl kickstart -k`).

### Trigger: the daily trading cycle starves the event loop

`httpx.ReadTimeout` means the TCP connection was ACCEPTED but no HTTP response arrived within 10s — the process is alive, its asyncio event loop is not serving. The alert window ~20:05-20:50 CEST sits exactly inside the daily cycle, which starts at 20:00-20:02 CEST every trading day (backend.log "Step 3 -- Analyzing ..." lines at 20:00:41/20:00:45/20:01:07/.../20:02:06 CEST for 05-28 through 06-10) and ends 21:02-21:34 CEST (`strategy_decisions` heartbeats 18:12-19:34 UTC).

This is a **known, previously diagnosed failure mode** — the code says so itself: `backend/services/autonomous_loop.py:716-723` (phase-23.1.23 comment: mark_to_market's blocking network ops "previously blocked /api/health past the watchdog threshold and got the backend kickstart-killed daily"). The mitigation wrapped *some* blocking calls in `asyncio.to_thread`, but blocking work remains on the loop: the lite analyzer calls `yf.Ticker(ticker)` / `stock.info` / `stock.history()` synchronously inside `async def _run_claude_analysis` (`autonomous_loop.py:1492-1494`; same pattern in `_run_gemini_analysis` at `:1744-1746`), and the cycle additionally ran full-orchestrator attempts that block/hang for 90s per agent before timing out (section 4). Several such calls back-to-back exceed the 10s probe budget repeatedly across the cycle window.

### Was the backend actually down?

**No — on 06-04 categorically not.** Restart census from backend.log (every `faulthandler registered on SIGUSR1` line = process start, date-anchored via APScheduler timestamps):

```
2026-05-27 18:59:25, 19:48:52, 21:03:49
2026-05-28 20:39:31, 23:51:56
2026-05-29 00:23:20, 22:47:44, 22:48:31, 23:11:57
2026-06-01 01:42:42
(next) 2026-06-11 05:03:45
```

- **06-04:** zero restarts — the backend ran uninterrupted from 06-01 01:42 to 06-11 05:03 (the whole away week). The 06-04 cycle completed normally: trades at 21:00-21:01 CEST, heartbeat 21:02 CEST, snapshot written. The alert was pure event-loop starvation, followed by the scheduler's recovery transition after the cycle.
- **05-27 / 05-28 (pre-away, operator at home):** the alerts bracket *mid-cycle graceful SIGTERM restarts* — 05-27 19:48:49 "Shutting down" while Deep Dive/Synthesis for MU/KEYS/GEV were in flight, immediately followed at 19:49:01 by a `POST /api/paper-trading/run-now` from localhost (a human at the dashboard); 05-28 20:39:27 "Shutting down" mid-cycle (39 min into the 20:00 cycle, stale lock age 2370s confirmed at restart). Both are clean SIGTERM shutdowns, not crashes and not bash-watchdog kills.
- **The bash watchdog never fired in this period:** zero SIGUSR1 stack dumps in backend.log after 2026-05-04 (6 dumps total, all 04-30..05-04), and no kickstart-pattern (SIGKILL = no graceful shutdown lines) restarts in the window.

### Verdict

Trigger identified: scheduled daily-cycle event-loop starvation (residual sync I/O on the loop, worsened that week by full-orchestrator 90s hang-then-timeout loops), plus operator/dev mid-cycle restarts on 05-27/05-28. The backend never crashed and no cycle was lost in the away week. The alert is doing its job, but it cannot distinguish "down" from "busy"; the operator read it as instability. Severity: P2 for the alert semantics; the underlying starvation is P1 because it also kept the bash watchdog one threshold away from SIGKILLing a live trading cycle (which is exactly what the 05-27 19:48 restart did manually — see incident 3).

## 3. Incident: 05-28 morning digest — ALL analyses 0.0/10 HOLD

### What the digest shows and where the zeros came from

The morning digest renders the latest 5 rows of `/api/reports/` (`scheduler.py:404`). As of 05-28 08:00 ET, the latest 5 `analysis_results` rows were the tail of the 05-27 evening re-run — all 0.0/HOLD. Primary rows (`financial_reports.analysis_results`, UTC):

```
05-27 17:18-17:36  STX 5.35 Sell | AMD 6.12 | CIEN 5.52 | HPE 6.42     (first run, scored)
05-27 18:02-18:20  STX 0.0 HOLD | HPE 0.0 | GEV 0.0 | MU 0.0 | KEYS 0.0
                   INTC 0.0 | ON 0.0 | DELL 0.0 | SNDK 0.0 | WDC 0.0 | GLW 0.0   (re-run, 12 of 13 zero)
```

### Root cause chain (file:line, log-verified)

1. **19:48:49 CEST: mid-cycle SIGTERM.** The 05-27 cycle (started 18:59 CEST) was killed mid-flight by a graceful shutdown (backend.log: "Shutting down" at 19:48:49 while `claude_code_invoke` calls for MU/GEV/KEYS were in progress); restart at 19:48:52; stale cycle-lock recovery (`prior_cycle_id=cycle-1779901170, age_sec=2962`); re-run triggered at 19:49:01 via `POST /api/paper-trading/run-now`.
2. **After the restart, every `claude` CLI invocation failed.** backend.log, repeating for every Claude-tier call in the re-run:
   ```
   20:02:12 E [claude_code_client] claude_code_invoke: non-zero exit code=1 stderr=
   20:02:12 W [claude_code_client] ClaudeCodeClient: generate_content failed (ClaudeCodeError('claude CLI exited with code 1: ')); returning empty LLMResponse
   ```
   Raised at `backend/agents/claude_code_client.py:190-197`. The configured rail was `paper_use_claude_code_route=True` with standard model `claude-sonnet-4-6`, deep-think `claude-opus-4-7` (startup log lines 19:48:52). The Gemini sub-agents kept working (88 `gemini-2.0-flash` rows in `llm_call_log` on 05-27), so each ticker produced a full enrichment context and then hit a dead Claude synthesis tier.
3. **The orchestrator degrades instead of failing.** Synthesis draft → empty; Critic invalid JSON → "treating as PASS with draft" (`backend/agents/orchestrator.py:1399`); Synthesis-Final invalid JSON → "Failed to parse final report, returning error" and returns `{"error": ...}` (`orchestrator.py:1451-1452`); risk-debate analysts invalid JSON → raw-text fallback; the pipeline then computes `final_json["final_weighted_score"] = compute_weighted_score({})` = **0.0** over the empty scoring matrix (`orchestrator.py:2000`) and returns a "successful" report.
4. **The autonomous loop maps the empty synthesis to 0.0/HOLD.** `autonomous_loop.py:1374-1394`: `rec = synthesis.get("recommendation", {})` → `{}` → `rec.get("action", "HOLD")` = "HOLD"; `final_score = synthesis.get("final_weighted_score", ...)` = 0.0. `_persist_analysis` (`:1904-1944`) writes the row with no score-validity check ("Non-fatal: any BQ error logs a warning but the trading cycle continues" — and no non-BQ checks at all).
5. **Display layer has no anomaly guard.** The digest formatter prints whatever score the row carries.

**Honest bound on the CLI failure itself:** exit code 1 with *empty stderr* is all the logs hold (the client truncates and stderr was genuinely empty; stdout is not logged on the error path). The failure began immediately after the 19:48:52 restart and had recovered by the 05-28 18:02 UTC cycle (which produced scored rows on the same rail). Consistent candidates are a Max-subscription session/usage-limit window or transient CLI auth/lock state inherited by the restarted launchd process; the data cannot discriminate between them. What is NOT open: every downstream layer turned a hard rail outage into silent 0.0/HOLD rows.

### Why nothing alerted

- No guard between "LLM returned nothing" and "score=0.0 persisted": `claude_code_client` returns an empty `LLMResponse` instead of raising at the orchestrator boundary; the critic treats unparseable output as PASS (`orchestrator.py:1399`); the error report still gets a numeric score (`:2000`).
- The cycle-completion Slack alert is informational (P3 `cycle_completed_summary`, `autonomous_loop.py:1286-1300`) and carries trade counts, not score sanity.
- The cost tracker NOTICED something was wrong-ish ("Cost budget exceeded: $4.3262 > $0.50 limit", `cost_tracker` warning 20:02:30) — log-only, not enforced, not alerted (AW-10).
- A 0.0 score is also a *legal* value in the schema; nothing distinguishes "scored 0" from "scoring failed". The phase-56.2 degraded-scoring guard (all-0.0 cycle → Slack alert) is precisely the missing control; as of the snapshot it does not exist (no such check in `autonomous_loop.py`, `scheduler.py`, or `formatters.py`).

Note: the earlier all-0.0 episodes (2026-05-22..05-26) had a *different* root cause — the `final_score` vs `final_weighted_score` key bug, fixed in commit `b9a1b772` (2026-05-26 19:05). The 05-27 zeros are post-fix and mechanically distinct (rail outage, not key mismatch). Two different bugs producing the identical silent symptom within one week is itself evidence that the missing control is the *guard*, not the individual bug fixes.

## 6. Reasoning-quality spot-checks (3+ stored analyses, including whipsaw trades)

### Spot-check A — MU 06-08 BUY → 06-09 forced exit (the −6.3% one-day round trip; whipsaw trade)

- **The entry reasoning** (`analysis_results` 06-08 18:10 UTC, BUY 7.0, conf 68): "All momentum and market-cap gates clear with no clear negative signal, though the +124% 60-day run and 45x P/E warrant a tight trailing stop against mean-reversion risk." - **The exit had no reasoning at all.** There is **no MU analysis row on 06-09**. The 06-09 cycle analyzed only 5 screener candidates (066570.KS, 009150.KS, 000660.KS, DELL, HPE) and `0 re-evals` (backend.log Step-3 line 06-09: "Analyzing 5 new + 0 re-evals"). The mechanism: `execute_buy` stamps `last_analysis_date=now` (`paper_trader.py:303,327`); re-eval only triggers at `days_since >= paper_reeval_frequency_days` = 3 (`autonomous_loop.py:727-740`, `settings.py:308`); the swap path then treats any holding absent from the same-cycle `holding_lookup` as **conviction 0.0** — the code says it plainly: "No fresh analysis => unknown conviction. **Treat as worst**" (`portfolio_manager.py:443-449`). Swap delta vs a 0.0 sentinel = `cand_score/0.01*100` (`:497-498`) — astronomically past the 25% threshold for any candidate. So MU, bought 24 hours earlier at the cycle's top score, was sold as "swap_for_higher_conviction" for a candidate with the *same* score (DELL 7.0). The label is false: it was swap-vs-unknown. The identical mechanism produced the STX/DELL exits on 06-08 (bought 06-05, 2d23h < 3 days, sentinel), the SNDK exit on 06-09, and the DELL exit on 06-10 — the away week's churn engine. **Verdict: the cited skill outputs did NOT inform the exit decision; no decision-quality reasoning existed for the money-losing leg.** (Also note `_compute_swap_candidates`'s own comment believes "final_score lives in [0,1]" (`portfolio_manager.py:495-496`) while the lite path emits 1-10 — the 25% relative threshold is mis-scaled by an order of magnitude even for *fresh* holdings: any +2 integer-point difference (7 vs 5) = +40% "uplift" and fires.)
