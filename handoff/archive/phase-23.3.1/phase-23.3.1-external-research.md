### Recency scan (2024-2026)

Searched for 2024-2026 literature on APScheduler job naming, id parameters, and replace_existing. Result: No new findings specifically about job-naming conventions in 2024-2026. APScheduler 3.11.2 (released in the 3.x series, current stable in that branch) has not changed the add_job signature for id/name/replace_existing since 3.6. The only relevant 2024-2026 development is APScheduler 4.x (stable as of ~2024) which renames add_job to add_schedule for recurring jobs and changes job.id from string UUID to a proper uuid.UUID type — this does NOT affect pyfinagent since the installed version is 3.11.2 and migration is a separate decision. The anonymous-UUID footgun (agronholm/apscheduler#487) was filed in 2021 and remains unresolved in 3.x as of 2026.

---

### Key findings

1. **UUID auto-generation on missing id** -- APScheduler 3.x generates a UUID4 string id when no `id=` is supplied. The value is non-deterministic per process start. (Source: apscheduler.readthedocs.io/en/3.x/modules/schedulers/base.html, accessed 2026-05-05)

2. **name fallback to qualname** -- When no `name=` is supplied, APScheduler sets `job.name` to the callable's `__qualname__`. For closures defined inside an async lifespan function, this produces noisy names like `lifespan.<locals>.process_batch`. (Source: APScheduler 3.x source behavior, confirmed via live observation per main session context)

3. **replace_existing=True is mandatory for persistent stores, best-practice for in-memory** -- The official user guide states explicitly: "you MUST define an explicit ID for the job and use replace_existing=True or you will get a new copy of the job every time your application restarts." (Source: apscheduler.readthedocs.io/en/3.x/userguide.html, accessed 2026-05-05)

4. **Dashboard description field = job.name or job.id** -- cron_dashboard_api.py:143 confirms: `"description": getattr(job, "name", None) or getattr(job, "id", "?")`. Explicit `name=` is the cleanest fix; it takes priority over the id fallback. (Source: internal audit, backend/api/cron_dashboard_api.py:143)

5. **Anonymous UUID breaks log correlation** -- GitHub issue #487 (agronholm/apscheduler) documents that without consistent id + name, the three lifecycle log events (adding / running / removing) reference different identifiers, making it impossible to correlate them in operator logs. (Source: github.com/agronholm/apscheduler/issues/487, accessed 2026-05-05)

6. **APScheduler 4.x migration note** -- add_job() in 4.x returns a uuid.UUID and is intended for one-off jobs; recurring jobs use add_schedule(). pyfinagent is on 3.11.2 and this is out of scope for phase-23.3.1, but the generator should note it for any future upgrade. (Source: apscheduler.readthedocs.io/en/master/migration.html, accessed 2026-05-05)

### Consensus vs debate

**Consensus**: All sources agree that explicit `id=` + `replace_existing=True` is required for any production job that must survive restarts, especially with persistent stores. The pattern is mandated in the official user guide, evidenced in multiple community issues, and already adopted in paper_trading.py.

**Debate**: Whether `name=` is strictly necessary vs just `id=` for dashboard readability. The `_job_to_dict` fallback (`job.name or job.id`) means a clean `id=` string alone improves the Jobs tab, but an explicit `name=` decouples the display label from the internal key — better practice for long-lived production code.

---

### Pitfalls (from literature)

1. **Duplicate job accumulation** -- Missing `replace_existing=True` with a persistent store creates a new job row on every restart. With in-memory stores this silently works on the current process but is a trap if the store ever migrates. (flask-apscheduler/issues/7)

2. **Log correlation breakage** -- UUID ids in log lines are unlinkable without grepping the scheduler's startup log. Named ids allow `grep paper_trading_daily` to pull the full lifecycle. (agronholm/apscheduler#487)

3. **APScheduler 4.x API break** -- add_job semantics change significantly in 4.x (recurring vs one-off split). If pyfinagent ever upgrades, both call sites (main.py:217 and paper_trading.py:914) need revisiting. Mitigation: use the 3.x string id convention now; 4.x uses uuid.UUID, but explicit string conversion is supported.
