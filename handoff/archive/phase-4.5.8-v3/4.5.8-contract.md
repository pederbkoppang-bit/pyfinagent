# Step 4.5.8 Contract -- Signal-freshness + cycle-health strip

**Drafted:** 2026-04-16
**Parent:** `handoff/current/phase-4.5-contract.md`
**Research tier:** simple (phase-level contract cites Grafana oncall pattern)

## Hypothesis
A per-source freshness strip (last_tick_age, BQ lag) plus a cycle-history log of the last 10 autonomous-loop runs gives operators the SLA signal they need to detect silent outages. Crucially, the dashboard must not rely *solely* on BQ-row timestamps (the dead-man's-switch failure mode: when the emitter dies, the dashboard shows OK because there is no new data to invalidate the last value).

## Success criteria (from masterplan 4.5.8)
1. last_tick_age_per_data_source_displayed -- per-source age for `paper_trades`, `paper_snapshots`, `paper_cycle_history` (local JSONL).
2. loop_heartbeat_alert_when_gt_2x_cycle_interval -- critical at 2x; amber warning at 1.5x (two-tier per Memfault / OneUptime watchdog convention).
3. bigquery_ingest_lag_shown -- derived from `MAX(event_time)` on the data table (Metaplane Method 1), NOT `INFORMATION_SCHEMA.TABLES.last_change_time` (Metaplane anti-pattern: schema changes corrupt the signal).
4. cycle_history_endpoint_returns_last_10_runs_with_timings -- Oracle Analytics Cloud canonical field set: cycle_id, started_at, completed_at, duration_ms, status, error_count, n_trades.
5. evaluator_critique_pass.

## Research findings (full entry appended to RESEARCH.md)

- **Freshness thresholds (Conduktor/dbt Labs/Monte Carlo consensus):** green `< 1x` expected interval, amber `1x-2x`, red `> 2x`. Amber at ~1.5x = "warning"; red at 2x = SLA breach.
- **Heartbeat factor (Memfault/OneUptime/Industrial Monitor Direct):** 2x is the modal single-threshold critical value; two-tier with warn@1.5x + critical@2x is more robust.
- **BQ lag (Metaplane):** use `SELECT MAX(event_time) FROM table` (Method 1); `last_change_time` is unreliable (schema migrations flip it). `STREAMING_TIMELINE` reports 1-minute windows, too coarse.
- **Cycle-history fields (Oracle OAC + GitHub Actions + Databricks):** cycle_id, started_at, completed_at, duration_ms, status, error_count, n_trades, data_source_ages_json, bq_ingest_lag_sec.
- **Anti-pattern: dead-man's-switch silent outage (Prometheus/Alertmanager):** empty != zero; when the emitter dies, absence-of-data can be read as "metric unchanged". Correct pattern: emit a *constant heartbeat from the emitter process itself* to an external relay; alert on absence. Translates here to a control-plane file `handoff/.cycle_heartbeat.json` updated each cycle, so the freshness endpoint checks both (a) BQ row age (data plane) and (b) heartbeat file age (control plane).

## Design

**Backend (`backend/services/cycle_health.py` -- new):**
- `CycleHealthLog`: thread-safe JSONL append-only at `handoff/cycle_history.jsonl`; retains unlimited history, reads tail for last N. Why file and not BQ: avoids blocking on a BQ schema migration to ship 4.5.8; the same pattern kill_switch uses.
- `record_cycle_start(cycle_id) -> started_at` and `record_cycle_end(cycle_id, status, duration_ms, error_count, n_trades, data_source_ages, bq_ingest_lag_sec)`.
- Heartbeat file `handoff/.cycle_heartbeat.json` with `{cycle_id, updated_at}`; written on start + end (so mid-cycle crash shows a stale heartbeat independent of BQ writes).
- `last_cycles(n=10) -> list[dict]` reads tail.
- `compute_freshness(bq, cycle_interval_sec)` -> dict with per-source ages, heartbeat ratio, bq_ingest_lag, color bands.

**Autonomous loop hooks:**
- At top of `run_daily_cycle`: `record_cycle_start(cycle_id)`.
- In `finally`: `record_cycle_end(...)` with status=success/failed/halted and error_count derived from `summary["errors"]` if present.

**Endpoints (`backend/api/paper_trading.py`):**
- `GET /cycles/history?limit=10` -> list of last N cycles.
- `GET /freshness` -> {sources: {name: {last_tick_age_sec, ratio, band}}, heartbeat: {age_sec, ratio, band}, bq_ingest_lag_sec, thresholds: {warn_ratio, critical_ratio}}.

**Frontend (`frontend/src/components/CycleHealthStrip.tsx` -- new):**
- Row of color-coded pills: one per source + heartbeat + BQ lag.
- Colors: emerald (green), amber, rose (red) driven by `band` field from backend (not hardcoded in UI).
- Below pills: compact table of last 10 cycles (id short, start time, duration, status, error_count, n_trades).
- Polls `/freshness` every 30s (sub-cycle cadence), `/cycles/history` every 60s.

**Thresholds (in response payload per contract §research point 1):**
- `warn_ratio = 1.5`
- `critical_ratio = 2.0`

## Anti-patterns guarded
- **Dead-man's-switch**: the heartbeat file is written by autonomous_loop itself, so a loop crash makes the heartbeat stale even if BQ is fine. Freshness endpoint reports both planes.
- **Schema-migration false-fresh**: BQ lag uses `MAX(event_time)` on the data column, never `INFORMATION_SCHEMA.TABLES.last_change_time`.
- **Color hardcoded in UI**: UI reads `band` from payload so operators can tune thresholds via settings without a frontend re-deploy.

## Out of scope
- BQ-backed cycle history table (file-based is sufficient for paper trading; upgrade later).
- Slack/email paging when a cycle breaches SLA (separate monitoring track).
- Per-signal-source freshness for macro/fundamentals (step is about cycle + BQ ingest).

## Risk
- File-based JSONL won't replicate across backend replicas. Acceptable for paper trading; document for multi-replica migration.
- Heartbeat file write must not block the cycle; wrap in try/except.
